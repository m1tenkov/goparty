# Настройка сервера и соединений

Проект GoParty реализован как серверное приложение для обработки событий от сообщества ВКонтакте. Основная серверная часть построена вокруг FastAPI-приложения, которое принимает входящие callback-запросы от VK, передает их во внутреннюю очередь обработки и далее запускает прикладную логику бота.

## Общая схема работы

На сервере запускается Python-приложение `app.py`. Оно поднимает FastAPI-сервер и предоставляет два HTTP-маршрута:

- `POST /vk/callback` - основной endpoint для событий VK Callback API;
- `GET /health` - технический endpoint для проверки доступности приложения.

События от VK не обрабатываются напрямую в момент приема запроса. После проверки callback-secret полезная нагрузка помещается во внутреннюю очередь из `event_processing.py`. Отдельный worker забирает события из очереди и передает их в обработчики бота. Такой подход нужен, чтобы endpoint быстро отвечал VK сообщением `ok` и не зависел от длительности пользовательского сценария.

## Соединение с VK

Бот работает через VK Callback API. Это означает, что не бот постоянно опрашивает VK, а VK сам отправляет HTTP-запросы на публичный адрес сервера при новых событиях.

В проекте обрабатываются основные типы событий:

- `confirmation` - подтверждение сервера при настройке Callback API;
- `message_new` - обычное входящее сообщение пользователя;
- `message_event` - нажатия inline-кнопок.

Для защиты callback-маршрута используется `VK_CALLBACK_SECRET`. При каждом callback-запросе приложение сравнивает поле `secret` из запроса VK со значением, заданным в переменной окружения или файле `secrets/vk_callback_secret.txt`. Для подтверждения сервера используется `VK_CALLBACK_CONFIRMATION_TOKEN`.

## HTTPS и SSL

FastAPI-приложение в проекте по умолчанию слушает локальный адрес `127.0.0.1` и порт `8000`. Само приложение не поднимает SSL-сертификат напрямую. Для публичного доступа к боту предполагается схема с reverse proxy:

1. VK отправляет запрос на публичный HTTPS-адрес вида `https://<domain>/vk/callback`.
2. HTTPS-соединение и SSL-сертификат обслуживает nginx или другой reverse proxy.
3. Reverse proxy перенаправляет запрос во внутреннее приложение на `http://127.0.0.1:8000`.

Таким образом, SSL относится к внешнему веб-соединению между VK и сервером. Его задача - обеспечить защищенную передачу callback-запросов по HTTPS. На уровне Python-приложения остается обычное локальное HTTP-соединение между nginx и uvicorn.

Для выпуска сертификата может использоваться Let's Encrypt. В этом случае nginx принимает HTTPS-трафик на 443 порту, а uvicorn остается закрытым от внешнего интернета и доступен только локально.

## TCP-соединение с базой данных

Данные бота хранятся в MySQL. Подключение выполняется через библиотеку `PyMySQL` по параметрам из `config.py`:

- `DB_HOST`;
- `DB_PORT`;
- `DB_NAME`;
- `DB_USER`;
- `DB_PASSWORD`;
- `DB_SSL_CA`.

По умолчанию используется порт `3306`, то есть стандартное TCP-соединение MySQL. В `database.py` создана обертка `ReconnectingConnection`, которая хранит подключение и перед выполнением запросов проверяет его через `ping(reconnect=True)`. Если соединение было разорвано, оно восстанавливается автоматически.

Для подключения к MySQL также предусмотрена опциональная SSL-настройка. Если задан параметр `DB_SSL_CA`, путь к CA-сертификату передается в `PyMySQL` как `ssl={"ca": ...}`. Это нужно для защищенного соединения с базой данных, например если MySQL находится не на том же сервере или требует SSL-подключение.

## SSH

SSH не используется внутри кода бота как часть пользовательского сценария или обмена сообщениями с VK. Его роль административная: через SSH можно подключаться к серверу, загружать код, редактировать конфигурацию, смотреть логи, перезапускать сервис и обслуживать nginx, MySQL или systemd-unit.

То есть SSH - это канал управления сервером, а не рабочий канал бота. Рабочие каналы приложения следующие:

- HTTPS от VK до публичного callback endpoint;
- локальный HTTP от reverse proxy до FastAPI/uvicorn;
- TCP-соединение от приложения до MySQL;
- HTTPS-запросы от приложения к VK API при отправке сообщений и получении данных.

## Хранение секретов и параметров

Конфигурация загружается из переменных окружения или из файлов в папке `secrets`. Для VK используются:

- `VK_BOT_TOKEN` или `secrets/token.txt`;
- `VK_CALLBACK_SECRET` или `secrets/vk_callback_secret.txt`;
- `VK_CALLBACK_CONFIRMATION_TOKEN` или `secrets/vk_callback_confirmation_token.txt`.

Параметры базы данных могут задаваться через переменные окружения или через JSON-файл `secrets/local_db.json`. Такой подход отделяет секреты от основного кода и позволяет переносить проект между локальной средой и сервером без изменения бизнес-логики.

## Запуск серверной части

Локально приложение может быть запущено командой:

```bash
uvicorn app:app --host 127.0.0.1 --port 8000
```

На сервере приложение целесообразно запускать как постоянный сервис, например через `systemd`. В такой схеме systemd следит за процессом uvicorn и перезапускает его при сбоях, а nginx принимает внешний HTTPS-трафик и проксирует его во внутреннее приложение.

Итоговая схема выглядит так:

```text
VK Callback API
    -> HTTPS / SSL
nginx на публичном сервере
    -> локальный HTTP
uvicorn + FastAPI app.py
    -> внутренняя очередь событий
event_processing.py
    -> логика бота
bot_handlers/*
    -> TCP / опционально SSL
MySQL
```

Такая архитектура разделяет публичный защищенный вход, внутреннюю обработку событий и хранение данных. Это упрощает настройку SSL, снижает риск прямого доступа к приложению uvicorn извне и позволяет отдельно управлять веб-сервером, Python-процессом и базой данных.
