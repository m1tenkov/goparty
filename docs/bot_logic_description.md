# Логика работы бота

GoParty реализован как VK-бот с серверной частью на FastAPI. Входящие события VK попадают в callback-обработчик, затем передаются во внутреннюю очередь и обрабатываются логикой из `bot_handlers`.

`app.py` не выполняет пользовательский сценарий синхронно внутри HTTP-запроса. Для событий `message_new` и `message_event` payload копируется во внутреннюю очередь, после чего VK сразу получает `ok`. Worker из `event_processing.py` создает VK API-клиент, преобразует payload в объект события и вызывает `handle_message(...)` или `handle_message_event(...)`. Если очередь заполнена, callback endpoint возвращает `503 Callback queue is full`.

## Регистрация анкеты

Пользователь последовательно указывает:

- имя;
- возраст;
- пол;
- город;
- кого ищет;
- игры;
- наличие микрофона;
- описание;
- фотографии.

Основные поля анкеты сохраняются в `profiles`, выбранные игры - в `user_games`, фотографии - в `user_photos`, а настройки поиска - в `user_filters` и `user_filter_games`.

При первом запуске бот пытается подтянуть доступные данные VK-профиля: имя, город, пол и возраст, если дата рождения содержит год. Пользователь может оставить предложенные значения или заменить их. Выбор игр в регистрации и выбор обязательных игр в фильтрах выполняются через inline callback-кнопки `message_event`.

Фотографии хранятся как локальные файлы в `storage/photos` и как VK photo token, если его удалось получить. В анкете используется до трех фотографий. При ошибках доставки сообщений VK профиль может быть помечен как `delivery_disabled`, чтобы не попадать в выдачу и не получать новые уведомления до следующего успешного обращения пользователя.

## Просмотр анкет

Подбор кандидата выполняется на основе:

- активности анкеты;
- отсутствия блокировки;
- доступности доставки сообщений;
- заполненности обязательных полей;
- пола и возрастного диапазона из фильтров;
- обязательных игр из фильтра и количества совпадающих игр;
- города и выбранного режима сортировки;
- предпочтения по микрофону;
- истории предыдущих действий.

История действий хранится в `interactions`. Если пользователь уже поставил кандидату лайк или дизлайк, эта анкета не показывается повторно в обычном режиме просмотра. Также из выдачи исключаются анкеты, которые уже поставили текущему пользователю дизлайк.

В фильтрах пользователь настраивает:

- кого ищет;
- сортировку `games` или `city`;
- возрастной диапазон;
- обязательные игры;
- требование наличия микрофона.

При сортировке `games` выше показываются анкеты с большим числом общих игр, затем анкеты из того же города и более близкие по возрасту. При сортировке `city` сначала учитывается совпадение города, затем общие игры и возраст.

Кнопка возврата к предыдущей анкете работает через историю `interactions`: бот показывает последний лайкнутый или дизлайкнутый профиль. Если пользователь смотрит историю и предыдущим действием был лайк, повторный лайк, дизлайк или лайк с сообщением блокируются, чтобы не перезаписать уже выбранное действие.

## Лайки, дизлайки и мэтчи

Все действия пользователя по отношению к другим анкетам сохраняются в таблице `interactions`.

Обычный лайк записывается как:

```text
from_user_id -> to_user_id, action = like
```

Дизлайк записывается как:

```text
from_user_id -> to_user_id, action = dislike
```

Если пользователь отправляет лайк с сообщением, текст сохраняется в поле `interactions.like_message`.

Входящий лайк без ответа больше не хранится в отдельной таблице. Он определяется как лайк, направленный текущему пользователю, для которого еще нет встречного действия от текущего пользователя. Мэтч также не хранится отдельно: он определяется по двум встречным лайкам.

Если включен `ENABLE_LIKE_NOTIFICATIONS`, получатель лайка сразу получает уведомление с анкетой отправителя. Ответ на входящий лайк может быть обычным лайком, лайком с сообщением, дизлайком или жалобой. При взаимном лайке бот отправляет уведомление о мэтче обеим сторонам, если доставку можно выполнить.

Пользователь с заблокированной анкетой не может лайкать других пользователей. При попытке взаимодействия с анкетой, которая стала неактивной, заблокированной или недоступной для доставки, бот сбрасывает текущего кандидата и показывает следующую подходящую анкету.

## Жалобы и модерация

Жалоба запускается из просмотра анкеты или из входящего лайка. Пользователь вводит текст причины до 500 символов без вложений. Бот отправляет карточку жалобы в модерационный чат, затем записывает для пользователя действие `dislike` в `interactions` и возвращает его к просмотру анкет.

## Редактирование и служебные действия

В меню анкеты пользователь может редактировать основные поля, описание, игры и фотографии, открыть фильтры, отключить анкету или сбросить данные, если соответствующая кнопка разрешена feature-флагом. Очистка истории удаляет только исходящие записи пользователя из `interactions`.

## Runtime-состояние

Состояние текущего диалога хранится в памяти приложения и дополнительно сохраняется в `user_sessions.session_json`. Это позволяет восстановить сценарий после перезапуска приложения или повторного обращения пользователя.

Ключевые runtime-поля:

- `step`
- `current_candidate`
- `browse_mode`
- `history_candidate_action`
- `history_cursor_id`
- `history_cursor_created_at`
- `pending_like_profile`
- `like_message_target_vk_user_id`
- `like_message_resume_step`
- `report_target_vk_user_id`
- `report_resume_step`
- `had_photos_before_edit`
- `resume_step`
- `suppress_post_match_prompt`

Постоянные пользовательские данные при этом остаются нормализованными в отдельных таблицах БД.
