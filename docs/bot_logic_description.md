# Логика работы бота GoParty

## Назначение

`GoParty` - VK-бот для знакомств и поиска напарников среди игроков. Пользователь проходит пошаговое заполнение анкеты, выбирает игры, добавляет описание и фотографии, а затем может:

- смотреть новые анкеты;
- возвращаться к истории уже просмотренных анкет;
- настраивать фильтры подбора;
- ставить лайк;
- отправлять лайк с сообщением;
- ставить дизлайк;
- отвечать на входящие лайки;
- жаловаться на анкеты;
- редактировать или временно отключать свою анкету;
- полностью сбрасывать анкету и историю.

## Архитектура обработки событий

Бот работает через `VK Callback API`.

Общий поток такой:

1. `app.py` принимает HTTP callback-событие от VK на `/vk/callback`.
2. Для `confirmation` сразу возвращается `VK_CALLBACK_CONFIRMATION_TOKEN`.
3. Для `message_new` и `message_event` payload кладется во внутреннюю очередь через `enqueue_callback_payload(...)`.
4. Worker, запущенный через `start_callback_worker()` при старте FastAPI, забирает события из очереди.
5. `event_processing.py` различает `message_new` и `message_event`.
6. Для текстовых сообщений вызывается `handle_message(...)`.
7. Для inline callback-событий вызывается `handle_message_event(...)`.
8. Диалоговая логика сосредоточена в `bot_handlers/router.py`.
9. Долговременные данные лежат в MySQL, а временный контекст сценария хранится и в памяти, и в `user_sessions`.

## Какие события реально используются

### `message_new`

Через обычные сообщения бот обрабатывает:

- старт и восстановление сценария;
- регистрацию;
- редактирование анкеты;
- настройку фильтров по полу, возрасту и микрофону;
- ввод описания;
- отправку фотографий;
- ввод сообщения к лайку;
- ввод причины жалобы;
- подтверждение отключения и сброса анкеты.

### `message_event`

Callback-события от inline-кнопок используются для:

- выбора игр в анкете;
- завершения шага выбора игр;
- выбора обязательных игр в фильтрах;
- очистки фильтра по играм;
- завершения настройки фильтра по играм.

## Восстановление пользователя

При новом сообщении бот:

1. создает запись в `users`, если она еще отсутствует;
2. подтягивает анкету, игры, сведения о фотографиях и фильтры из БД;
3. загружает `session_json`;
4. объединяет данные БД, сохраненную сессию и текущее состояние в памяти;
5. запрашивает открытые данные VK через `users.get`;
6. применяет автозаполнение имени, возраста, города и пола, если они еще не указаны.

Это делает `ensure_runtime_user(...)` вместе с `apply_vk_defaults(...)`.

## Регистрация и заполнение анкеты

Обязательные шаги:

1. имя;
2. возраст;
3. пол;
4. город;
5. кого ищет пользователь;
6. игры;
7. играет ли пользователь с микрофоном;
8. описание;
9. фотографии.

Особенности:

- возраст ограничен диапазоном `14-99`;
- описание ограничено `2000` символами;
- сообщение к лайку и причина жалобы ограничены `500` символами;
- можно хранить от `1` до `3` фотографий в локальном файловом хранилище, при этом в БД сохраняются пути к файлам;
- шаг выбора игр считается завершенным только после `games_done`;
- признак микрофона сохраняется в `profiles.uses_microphone` и отображается в анкете.

## Игры и inline-выбор

Поддерживаются игры:

- `dota2`
- `cs2`
- `minecraft`
- `mlbb`
- `valorant`
- `pubg`
- `dbd`
- `genshin`

Выбор игр идет через inline-кнопки. По callback:

- `toggle_game` переключает игру в runtime-состоянии;
- `games_done` сохраняет результат в `user_games` и ставит `games_step_completed = True`.

## Работа с фотографиями

В новой реализации фотографии не завязаны только на старые VK token.

Фактическая логика такая:

- бот извлекает фото из вложений сообщения;
- скачивает оригиналы или лучшие доступные размеры в локальное хранилище `storage/photos/<vk_user_id>/`;
- сохраняет файл на сервере, а в БД записывает `photo_path` и при необходимости `vk_photo_token`;
- при отправке анкеты повторно загружает локальные файлы в сообщения VK, если действительного токена нет;
- удаляет локальные файлы, которые больше не используются.

Это делает хранение фотографий устойчивее при редактировании анкеты и повторной отправке профиля.

## Экран своей анкеты

После завершения регистрации пользователь попадает в экран своей анкеты (`STATE_REVIEW`), где доступны:

- переход к просмотру анкет;
- переход к настройке фильтров;
- редактирование анкеты;
- временное отключение анкеты;
- полный сброс анкеты и истории, если включен флаг `ENABLE_PROFILE_RESET_BUTTON`;
- очистка истории, если включен флаг `ENABLE_CLEAR_HISTORY_BUTTON`.

## Редактирование анкеты

Редактирование разбито на разделы:

- основные поля;
- описание;
- игры;
- фотографии.

Изменения анкеты сохраняются без автоматического сброса ранее полученных дизлайков.

## Фильтры подбора

Это одно из главных новых расширений логики.

Пользователь может настроить:

- пол искомого пользователя;
- режим сортировки: по общим играм или с приоритетом анкет из того же города;
- возрастной диапазон;
- список обязательных игр, которые должны быть у кандидата;
- требование показывать только кандидатов с микрофоном.

Фильтры сохраняются в:

- `user_filters`
- `user_filter_games`

Во время подбора они влияют на SQL-запрос в `get_random_candidate(...)`.

## Подбор и просмотр анкет

Просмотр идет в состоянии `STATE_BROWSE`.

Кандидат:

- не должен быть самим пользователем;
- должен иметь активную анкету;
- не должен быть забанен;
- не должен иметь `delivery_disabled = 1`;
- должен иметь заполненные обязательные поля;
- не должен быть уже лайкнут или дизлайкнут текущим пользователем;
- не должен ранее дизлайкнуть текущего пользователя;
- должен соответствовать фильтрам по полу, возрасту, обязательным играм и микрофону, если они заданы.

Ранжирование кандидатов строится по:

1. числу общих игр;
2. совпадению города;
3. близости возраста;
4. времени создания профиля;
5. внутреннему `id`.

Если выбран режим сортировки `city`, приоритет города поднимается выше общих игр.

## История просмотра

Бот умеет возвращать пользователя к уже просмотренным анкетам через таблицу `interactions`.

Для этого используются:

- `browse_mode`
- `history_candidate_action`
- `history_cursor_id`
- `history_cursor_created_at`

Особенность истории:

- если анкета уже была лайкнута, повторно изменить оценку нельзя;
- в истории остается возможность пожаловаться и затем вернуться к новым анкетам.

## Лайки, дизлайки и мэтчи

### Обычный лайк

При лайке бот:

1. проверяет актуальность кандидата;
2. сохраняет `like` в `interactions`;
3. проверяет встречный лайк;
4. при совпадении создает запись в `matches`;
5. если мэтча нет, создает или обновляет `pending_likes`.

### Лайк с сообщением

Пользователь переходит в `STATE_LIKE_MESSAGE`, вводит текст до `500` символов, после чего:

- сохраняется обычный лайк;
- сообщение сохраняется в `pending_likes.like_message`;
- при уведомлении или мэтче это сообщение может быть показано второй стороне.

### Дизлайк

Дизлайк сохраняется как `dislike` в `interactions` и исключает анкету из дальнейшей выдачи.

## Входящие лайки

Если у пользователя есть необработанный `pending_like`, бот переводит его в `STATE_INCOMING_LIKE`.

Доступные действия:

- лайк;
- лайк с сообщением;
- дизлайк;
- жалоба.

После ответа:

- pending like помечается обработанным;
- бот либо показывает следующий входящий лайк;
- либо возвращает пользователя к предыдущему сценарию.

## Жалобы и модерация

Жалоба запускается для активного профиля и переводит пользователя в `STATE_REPORT_REASON`.

После ввода причины:

- бот отправляет сообщение в чат модерации;
- сохраняет отрицательное взаимодействие;
- закрывает соответствующий pending like, если он был;
- убирает текущую анкету из дальнейшей выдачи.

## Бан и доставка сообщений

### Бан

В `profiles` хранятся:

- `is_banned`
- `banned_at`
- `ban_reason`

Забаненный профиль:

- не участвует в выдаче;
- не может пользоваться сценарием как обычный пользователь;
- получает отдельное сообщение о блокировке.

### Недоставка сообщений

Если VK API возвращает ошибки доставки, профиль может быть помечен как недоступный:

- `delivery_disabled`
- `delivery_error_code`
- `delivery_error_at`

Такие профили исключаются из подбора. Как только пользователь снова пишет боту, `reset_profile_delivery_status(...)` сбрасывает этот статус.

## Деактивация и полный сброс

Пользователь может:

- временно отключить анкету через `STATE_DEACTIVATE_CONFIRM`;
- полностью удалить анкету и историю через `STATE_RESET_CONFIRM`.

После отключения анкета автоматически активируется при возврате в сценарий просмотра.

После полного сброса:

- удаляется пользователь и связанные данные по каскаду;
- очищаются локальные фотографии;
- runtime-состояние убирается из памяти;
- сценарий регистрации запускается заново.

## Защита от повторных действий

Для действий с высоким риском двойного срабатывания используется debounce-защита `is_duplicate_action(...)` с окном `0.5` секунды.

Она применяется к:

- лайкам;
- дизлайкам;
- лайкам с сообщением;
- ответам на входящие лайки;
- завершению выбора игр;
- переходам к просмотру анкет.

## Логирование

В проекте есть два основных лога:

- `logs/bot.log`
- `logs/actions.jsonl`

Логируются:

- входящие сообщения;
- исходящие сообщения;
- callback-события;
- ошибки отправки и редактирования;
- ошибки обработчиков;
- сохранение взаимодействий;
- итоговое состояние пользователя после обработки сообщения;
- длительность обработки событий.
