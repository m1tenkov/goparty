# Логика Работы Бота GoParty

## 1. Назначение проекта

`GoParty` — это VK-бот знакомств среди геймеров. Пользователь заполняет анкету, выбирает любимые игры, добавляет фото, после чего бот начинает показывать ему подходящие анкеты других пользователей. На каждую анкету можно поставить лайк, лайк с сообщением, дизлайк или пожаловаться. Если лайк взаимный, бот создает мэтч и сообщает об этом обеим сторонам.

## 2. Общая архитектура

Бот работает через `VK Long Poll`.

Основные части проекта:

- `main.py` — запускает бесконечный цикл прослушивания событий VK.
- `vk_bot.py` — создает VK-сессию и объект Long Poll.
- `bot_handlers/router.py` — основная логика диалога и состояний.
- `bot_handlers/utils.py` — вспомогательная логика для runtime-состояния, показа анкет, форматирования и работы с фото.
- `database.py` — слой доступа к MySQL/MariaDB.
- `bot_handlers/keyboards.py` — создание reply- и inline-клавиатур.
- `bot_handlers/texts.py` — все тексты, кнопки и шаблоны сообщений.
- `logger.py` — технические логи и журнал действий.

## 3. Какие события обрабатывает бот

Бот обрабатывает два основных типа событий:

- `MESSAGE_NEW` — обычное новое сообщение пользователя в диалоге с ботом.
- `MESSAGE_EVENT` — нажатие inline/callback-кнопок, в проекте это в первую очередь выбор игр.

В `main.py` каждое событие передается в `process_event(...)`, где:

1. определяется тип события;
2. извлекаются данные пользователя;
3. вызывается нужный обработчик;
4. логируется результат и длительность обработки.

Если Long Poll падает по ошибке, бот логирует проблему, ждет 3 секунды и создает новый объект Long Poll.

## 4. Хранение данных

### 4.1. Постоянные данные

Постоянные данные хранятся в базе данных:

- `users` — пользователи и их VK ID;
- `profiles` — основная информация анкеты;
- `games` — справочник игр;
- `user_games` — связь пользователя и выбранных игр;
- `user_photos` — список фотографий пользователя;
- `interactions` — лайки и дизлайки;
- `matches` — взаимные лайки;
- `pending_likes` — лайки, на которые еще не ответили;
- `user_sessions` — сохраненное runtime-состояние.

### 4.2. Runtime-состояние

У бота есть временное runtime-состояние пользователя. Оно нужно, чтобы бот понимал, где пользователь сейчас находится в диалоге.

Runtime-состояние хранится:

- в памяти процесса, в словаре `users`;
- в БД, в таблице `user_sessions`, чтобы состояние можно было восстановить после перезапуска.

В runtime-состоянии лежат, например:

- текущий шаг диалога `step`;
- текущая показываемая анкета `current_candidate`;
- режим просмотра `browse_mode`;
- временные данные для лайка с сообщением;
- временные данные для жалобы;
- информация о входящем лайке;
- служебные флаги возврата между сценариями.

## 5. Восстановление пользователя

Когда пользователь пишет боту, сначала вызывается `ensure_runtime_user(...)`.

Эта функция:

1. создает пользователя в `users`, если его еще нет;
2. загружает анкету из БД;
3. загружает сохраненное runtime-состояние из `user_sessions`;
4. подмешивает уже существующее состояние из памяти;
5. получает открытые данные VK-профиля для автозаполнения;
6. собирает единый runtime-объект пользователя.

Это позволяет не терять ход диалога после перезапуска бота.

## 6. Старт диалога

Старт пользовательского сценария происходит в `start_bot_flow(...)`.

Он вызывается в нескольких случаях:

- пользователь пишет боту впервые;
- пользователя нет в runtime-памяти;
- пользователь нажал кнопку `Начать`;
- пришел `payload` с командой `start`;
- после сброса анкеты бот запускает сценарий заново.

На старте бот:

1. проверяет, существовал ли пользователь раньше в БД;
2. восстанавливает runtime-состояние;
3. подтягивает данные из VK, если можно;
4. автоматически подставляет пустые поля из VK-профиля;
5. решает, что показывать дальше.

Если пользователь уже существовал в БД, бот отправляет сообщение `С возвращением!`.

## 7. Сценарий заполнения анкеты

Анкета заполняется пошагово. Бот ведет пользователя по обязательным шагам в функции `ask_next_required_field(...)`.

Порядок шагов:

1. имя;
2. возраст;
3. пол;
4. город;
5. кого ищет;
6. игры;
7. описание;
8. фотографии.

### 7.1. Имя

- бот ждет текст;
- имя не должно быть пустым;
- имя ограничено по длине;
- после успеха оно сохраняется в БД.

### 7.2. Возраст

- принимается только число;
- допустимый диапазон — от 14 до 99;
- после валидации возраст сохраняется.

### 7.3. Пол

Пользователь выбирает один из двух вариантов:

- `male`;
- `female`.

### 7.4. Город

- бот принимает текст;
- город должен быть непустым;
- есть ограничение по длине.

### 7.5. Кого ищет

Пользователь выбирает:

- `male`;
- `female`;
- `any`.

### 7.6. Игры

Игры выбираются не обычным текстом, а через inline callback-кнопки.

Поддерживаемые игры:

- Dota 2
- CS2
- Minecraft
- MLBB
- Valorant
- PUBG
- Dead by Daylight
- Genshin Impact

При выборе игр:

- бот меняет состояние кнопок;
- редактирует текущее inline-сообщение;
- после `Готово` сохраняет игры в `user_games`;
- помечает `games_step_completed = 1`.

### 7.7. Описание

- бот принимает только текст;
- вложения не допускаются;
- есть лимит длины.

### 7.8. Фото

Пользователь может загрузить от 1 до 3 фотографий.

Логика такая:

- если фото меньше 3, бот предлагает добавить еще;
- если фото больше 3, бот отправляет ошибку;
- фото сохраняются в `user_photos`;
- после завершения шага профиль считается заполненным по фото.

## 8. Экран своей анкеты

Когда анкета заполнена, бот переводит пользователя в `STATE_REVIEW`.

На этом экране показываются:

- текст анкеты;
- прикрепленные фото;
- клавиатура действий.

Основные действия с этого экрана:

- начать просмотр чужих анкет;
- изменить основные поля;
- изменить описание;
- изменить игры;
- изменить фотографии;
- отключить анкету;
- полностью сбросить данные.

## 9. Редактирование анкеты

У анкеты есть отдельное меню редактирования:

- имя;
- возраст;
- пол;
- кого ищет;
- город.

Описание, игры и фотографии редактируются отдельными сценариями.

После изменения важных полей бот дополнительно может очистить полученные ранее дизлайки, чтобы обновленная анкета снова могла участвовать в показах.

## 10. Просмотр анкет

Когда пользователь выбирает просмотр анкет, бот переводит его в `STATE_BROWSE`.

Подбор кандидата идет через `get_random_candidate(...)`.

Кандидат должен:

- не быть самим пользователем;
- быть активным;
- не быть забаненным;
- не иметь `delivery_disabled = 1`;
- иметь заполненные обязательные поля;
- не быть уже лайкнутым или дизлайкнутым текущим пользователем.

Также бот учитывает:

- пол текущего пользователя;
- кого он ищет;
- взаимные предпочтения;
- общие игры;
- близость возраста.

Сортировка идет по:

1. количеству общих игр;
2. близости возраста;
3. времени создания профиля.

## 11. История просмотра

Бот умеет возвращаться к предыдущим анкетам через историю взаимодействий.

Для этого используются:

- `browse_mode`;
- `history_candidate_action`;
- `history_cursor_id`;
- `history_cursor_created_at`.

История берется из таблицы `interactions`.

Если пользователь смотрит анкету из истории и ранее уже поставил ей лайк, бот не позволяет изменить лайк на что-то другое.

## 12. Лайки, pending likes и мэтчи

### 12.1. Обычный лайк

Когда пользователь нажимает лайк:

1. бот проверяет, что кандидат актуален;
2. проверяет защиту от двойного нажатия;
3. сохраняет действие в `interactions`;
4. проверяет, есть ли взаимный лайк.

Если лайк взаимный:

- создается запись в `matches`;
- бот отправляет уведомление о мэтче обеим сторонам.

Если лайк не взаимный:

- создается запись в `pending_likes`;
- при включенной настройке уведомлений второй пользователь получает уведомление, что его лайкнули.

### 12.2. Лайк с сообщением

Пользователь может не просто поставить лайк, а сначала перейти в сценарий `STATE_LIKE_MESSAGE` и ввести сообщение.

Дальше логика почти такая же:

- сохраняется `like`;
- текст попадает в `pending_likes.like_message`;
- при мэтче сообщение тоже может быть включено в уведомление.

### 12.3. Входящий лайк

Если пользователя кто-то лайкнул, бот может показать ему отдельный сценарий `STATE_INCOMING_LIKE`.

Пользователь может:

- лайкнуть в ответ;
- отправить ответный лайк с сообщением;
- поставить дизлайк;
- пожаловаться.

После ответа pending like помечается обработанным через `resolve_pending_like(...)`.

## 13. Жалобы

Если пользователь жалуется на анкету:

1. бот переводит его в `STATE_REPORT_REASON`;
2. ждет текст причины;
3. отправляет жалобу в чат модерации `peer_id = 2000000001`;
4. одновременно сохраняет `dislike` в `interactions`;
5. если был pending like, закрывает его как `dislike`.

В сообщение модерации включаются:

- анкета нарушителя;
- ссылка на жалующегося;
- ссылка на нарушителя;
- текст причины.

## 14. Блокировки анкет

В профиле пользователя есть поля:

- `is_banned`
- `banned_at`
- `ban_reason`

Забаненный пользователь:

- не должен попадать в выдачу другим;
- не должен отправлять рабочие лайки и лайки с сообщением;
- его лайки не должны приводить к уведомлениям другим пользователям.

Бан накладывается отдельным SQL-скриптом модерации.

## 15. Ошибки доставки сообщений

Если VK не позволяет отправить сообщение пользователю:

- бот фиксирует код ошибки;
- ставит `delivery_disabled = 1`;
- сохраняет `delivery_error_code`;
- сохраняет `delivery_error_at`.

Такие профили:

- исключаются из обычной выдачи;
- не должны получать стандартные уведомления.

Когда пользователь снова пишет боту, в начале обработки сообщения бот сбрасывает флаг недоставки через `reset_profile_delivery_status(...)`.

## 16. Отключение анкеты

Пользователь может вручную отключить свою анкету.

Для этого:

1. бот переводит его в `STATE_DEACTIVATE_CONFIRM`;
2. ждет подтверждение;
3. сохраняет `is_active = 0`.

Отключенная анкета не участвует в показах.

При этом, если пользователь позже снова нажимает `Смотреть анкеты`, бот автоматически реактивирует анкету через `reactivate_profile_if_needed(...)`.

## 17. Полный сброс данных

Если пользователь выбирает сброс:

1. бот переводит его в `STATE_RESET_CONFIRM`;
2. после подтверждения удаляет пользователя из `users`;
3. каскадно удаляются связанные данные;
4. runtime-состояние очищается;
5. сценарий запускается заново через `start_bot_flow(...)`.

## 18. Callback-обработка игр

Inline-кнопки выбора игр работают через `MESSAGE_EVENT`.

Логика:

- `toggle_game` — включает или выключает конкретную игру;
- `games_done` — завершает шаг выбора игр, сохраняет результат и переводит пользователя дальше.

После каждого callback бот:

- отвечает на событие VK;
- при необходимости редактирует inline-сообщение;
- обновляет runtime-состояние.

## 19. Защита от повторных нажатий

Для ряда действий бот использует debounce-защиту через `is_duplicate_action(...)`.

Она не дает слишком быстро повторно обработать:

- лайк;
- дизлайк;
- лайк с сообщением;
- ответ на входящий лайк;
- завершение выбора игр;
- переход в просмотр анкет.

Если действие считается дублем, бот его игнорирует и пишет это в лог.

## 20. Логирование

В проекте есть два основных вида логов:

- обычный файл `logs/bot.log`;
- структурированный журнал действий `logs/actions.jsonl`.

Логируются:

- входящие сообщения;
- исходящие сообщения;
- ошибки отправки;
- ошибки callback-ответов;
- сохранение взаимодействий;
- длительность обработки событий;
- изменения состояния пользователя.

Это нужно для отладки и анализа поведения бота.

## 21. Сквозной сценарий работы

Упрощенная последовательность работы выглядит так:

1. бот запускается;
2. подключается к VK API и БД;
3. проверяет и дополняет runtime-схему БД;
4. начинает слушать Long Poll;
5. получает сообщение пользователя;
6. восстанавливает или создает runtime-состояние;
7. если анкета не заполнена — ведет пользователя по шагам регистрации;
8. если анкета заполнена — показывает экран своей анкеты или входящий лайк;
9. после перехода к просмотру подбирает кандидата;
10. сохраняет лайки, дизлайки, жалобы и мэтчи в БД;
11. при перезапуске восстанавливает пользовательский сценарий из `user_sessions`.

## 22. Что важно помнить о проекте

- Бот работает не через `Callback API`, а через `Long Poll`.
- Основная логика диалога сосредоточена в `router.py`.
- Runtime-состояние — ключевая часть проекта, потому что именно оно позволяет продолжать диалог с нужного шага.
- База данных хранит не только анкеты, но и историю взаимодействий, pending likes и сохраненные пользовательские сессии.
- Сценарии лайков, мэтчей, жалоб и деактивации анкеты уже встроены в основную логику и не являются внешними модулями.
