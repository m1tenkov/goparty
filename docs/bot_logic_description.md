# Логика работы бота GoParty

## Назначение

`GoParty` — VK-бот для знакомств и поиска напарников среди игроков. Пользователь проходит пошаговое заполнение анкеты, указывает игры, добавляет описание и фотографии, после чего может:

- смотреть подходящие анкеты;
- ставить лайк;
- отправлять лайк с сообщением;
- ставить дизлайк;
- отвечать на входящие лайки;
- жаловаться на анкеты;
- редактировать, отключать или полностью сбрасывать свою анкету.

Если лайк становится взаимным, бот создает мэтч и отправляет уведомления обеим сторонам.

## Общая архитектура

Бот работает через `VK Long Poll`.

Основной поток такой:

1. `main.py` запускает бесконечное прослушивание событий VK.
2. Для `MESSAGE_NEW` вызывается `handle_message(...)`.
3. Для `MESSAGE_EVENT` вызывается `handle_message_event(...)`.
4. Логика диалога сосредоточена в `bot_handlers/router.py`.
5. Состояние пользователя хранится одновременно:
   - в памяти процесса, в словаре `bot_handlers.constants.users`;
   - в базе данных, в таблице `user_sessions`.
6. Данные анкеты, лайки, мэтчи, фотографии и история взаимодействий хранятся в MySQL.

## Какие события обрабатываются

### `MESSAGE_NEW`

Это обычные входящие сообщения пользователя. Через них бот обрабатывает:

- регистрацию;
- редактирование анкеты;
- отправку фотографий;
- просмотр анкет;
- лайки, дизлайки и жалобы;
- подтверждение деактивации и сброса.

### `MESSAGE_EVENT`

Это callback-события от inline-кнопок. В текущей версии они используются для выбора игр:

- `toggle_game` — переключение конкретной игры;
- `games_done` — завершение шага выбора игр.

После каждого callback бот отвечает через `messages.sendMessageEventAnswer`, а при необходимости редактирует исходное inline-сообщение.

## Конфигурация и запуск

При старте:

- `config.py` читает токен VK из `VK_BOT_TOKEN` или `secrets/token.txt`;
- `config.py` собирает конфиг БД из переменных окружения или `secrets/local_db.json`;
- `vk_bot.py` создает VK-сессию и объект Long Poll;
- `database.py` открывает MySQL-подключение и вызывает `ensure_runtime_schema()`.

`ensure_runtime_schema()` гарантирует наличие:

- таблицы `pending_likes`;
- таблицы `user_sessions`;
- служебных колонок в `profiles`:
  - `is_banned`
  - `banned_at`
  - `ban_reason`
  - `delivery_disabled`
  - `delivery_error_code`
  - `delivery_error_at`
  - `games_step_completed`

Если Long Poll падает по ошибке, `main.py` пишет ошибку в лог, ждет 3 секунды и пересоздает объект Long Poll.

## Хранение данных

### Постоянные данные

В базе используются следующие сущности:

- `users` — внутренние пользователи и их `vk_user_id`;
- `profiles` — основные поля анкеты;
- `games` — справочник игр;
- `user_games` — выбранные пользователем игры;
- `user_photos` — до 3 фотографий пользователя;
- `interactions` — лайки и дизлайки;
- `matches` — взаимные лайки;
- `pending_likes` — входящие лайки, на которые еще не ответили;
- `user_sessions` — сериализованное runtime-состояние пользователя.

### Runtime-состояние

В runtime-состоянии бот хранит то, что нужно для продолжения сценария без потери контекста. Например:

- `step` — текущий шаг диалога;
- `current_candidate` — текущая показываемая анкета;
- `browse_mode` — режим просмотра (`new` или `history`);
- `history_candidate_action`, `history_cursor_id`, `history_cursor_created_at` — курсор просмотра истории;
- `pending_like_profile` — профиль пользователя, который лайкнул текущего пользователя;
- `like_message_target_vk_user_id`, `like_message_resume_step` — состояние сценария лайка с сообщением;
- `report_target_vk_user_id`, `report_resume_step` — состояние сценария жалобы;
- `resume_step` — точка возврата после входящего лайка;
- `had_photos_before_edit` — флаг редактирования фото;
- `suppress_post_match_prompt` — служебный флаг после мэтча;
- `games_step_completed` — завершен ли шаг выбора игр.

## Восстановление пользователя

При любом новом сообщении бот сначала приводит пользователя к актуальному состоянию:

1. создает запись в `users`, если ее еще нет;
2. подтягивает анкету из `profiles`, игры из `user_games` и фото из `user_photos`;
3. загружает `session_json` из `user_sessions`;
4. объединяет состояние из БД и памяти;
5. запрашивает открытые данные VK-профиля через `users.get`;
6. сохраняет итоговый runtime-объект в `users`.

Это делает функция `ensure_runtime_user(...)`.

## Автозаполнение из VK

Бот умеет подставлять часть данных из профиля VK, если пользователь их еще не вводил:

- имя;
- возраст, если дата рождения содержит год;
- город;
- пол.

Автоподстановка выполняется в `apply_vk_defaults(...)` и записывается в БД.

## Сценарий старта

Старт происходит через `start_bot_flow(...)`.

Что делает бот:

1. восстанавливает runtime-состояние;
2. проверяет бан;
3. подставляет дефолты из VK;
4. повторно синхронизирует состояние с БД;
5. определяет, что показать дальше.

Дальнейшее поведение:

- если анкета уже полная и есть необработанные входящие лайки, бот сначала показывает входящий лайк;
- если анкета уже полная и входящих лайков нет, бот показывает экран своей анкеты;
- если анкета неполная, бот переводит пользователя на следующий обязательный шаг;
- если обязательные шаги уже пройдены, но есть pending like, бот показывает его;
- иначе показывает экран своей анкеты.

## Состояния диалога

В проекте используются следующие основные состояния:

- `REG_NAME`
- `REG_AGE`
- `REG_GENDER`
- `REG_CITY`
- `REG_LOOKING`
- `STATE_GAMES`
- `STATE_ABOUT`
- `STATE_PHOTO`
- `STATE_PHOTO_MORE`
- `STATE_PHOTO_APPEND`
- `STATE_REVIEW`
- `STATE_EDIT_MENU`
- `STATE_BROWSE`
- `STATE_INCOMING_LIKE`
- `STATE_LIKE_MESSAGE`
- `STATE_REPORT_REASON`
- `STATE_DEACTIVATE_CONFIRM`
- `STATE_RESET_CONFIRM`

## Пошаговое заполнение анкеты

Бот ведет пользователя по обязательным полям в `ask_next_required_field(...)`.

Порядок шагов:

1. имя;
2. возраст;
3. пол;
4. город;
5. кого ищет;
6. игры;
7. описание;
8. фотографии.

### Имя

- принимается только осмысленный текст;
- пустые и слишком длинные значения отклоняются;
- сохраняется через `save_profile_fields(...)`.

### Возраст

- принимается только число;
- допустимый диапазон: от 14 до 99;
- при наличии возраста из VK бот может предложить его кнопкой.

### Пол

Доступны варианты:

- `male`;
- `female`.

### Город

- принимается текст;
- пустые и слишком длинные значения отклоняются;
- при наличии города из VK бот может предложить его кнопкой.

### Кого ищет

Доступны варианты:

- `male`;
- `female`;
- `any`.

### Игры

Шаг работает не текстом, а через inline-кнопки. Поддерживаемые коды игр:

- `dota2`
- `cs2`
- `minecraft`
- `mlbb`
- `valorant`
- `pubg`
- `dbd`
- `genshin`

На шаге выбора игр:

- бот держит состояние выбора прямо в runtime-полях пользователя;
- по `toggle_game` переключает значение конкретной игры;
- по `games_done` сохраняет выбранные игры в `user_games`;
- помечает `games_step_completed = True`;
- редактирует inline-сообщение с кратким итогом.

### Описание

- принимается только текст;
- вложения не допускаются;
- лимит: 2000 символов;
- текст очищается от невидимых управляющих символов.

### Фотографии

Пользователь может сохранить от 1 до 3 фотографий.

Логика:

- бот извлекает только photo-вложения;
- другие типы вложений не сохраняются;
- если фото больше 3, бот отклоняет сообщение;
- после первой загрузки можно прислать еще фото или нажать кнопку завершения;
- фотографии сохраняются в `user_photos` в фиксированном порядке.

## Когда анкета считается полной

Анкета считается заполненной, если есть:

- `name`;
- `age`;
- `gender`;
- `city`;
- `looking_for`;
- выбранный шаг игр;
- `about`;
- хотя бы одна фотография.

Проверка выполняется в `is_profile_complete(...)`.

## Экран своей анкеты

После заполнения анкеты бот переводит пользователя в `STATE_REVIEW`.

На этом экране пользователь видит:

- текст своей анкеты;
- прикрепленные фото;
- reply-клавиатуру действий.

Отсюда доступны действия:

- смотреть анкеты;
- изменить базовую информацию;
- изменить описание;
- изменить игры;
- изменить фото;
- отключить анкету;
- полностью сбросить данные.

## Редактирование анкеты

Редактирование разбито на несколько сценариев:

- базовые поля открываются через `STATE_EDIT_MENU`;
- описание редактируется через `STATE_ABOUT`;
- игры — через `STATE_GAMES`;
- фото — через `STATE_PHOTO` и `STATE_PHOTO_MORE`.

При изменении важных данных бот очищает полученные ранее дизлайки пользователя. Это происходит для:

- имени;
- возраста;
- города;
- описания;
- пола;
- набора игр;
- фотографий.

Так обновленная анкета снова может полноценно участвовать в выдаче.

## Просмотр чужих анкет

Просмотр работает в `STATE_BROWSE`.

Бот подбирает кандидата через `get_random_candidate(...)`. Кандидат должен:

- не быть самим пользователем;
- иметь активную анкету;
- не быть забаненным;
- не иметь `delivery_disabled = 1`;
- иметь заполненные обязательные поля;
- не быть уже лайкнутым или дизлайкнутым текущим пользователем;
- не иметь ранее поставленного дизлайка текущему пользователю.

Дополнительно учитываются:

- пол, который ищет текущий пользователь;
- взаимная совместимость по полу;
- число общих игр;
- близость возраста.

Сортировка идет по:

1. количеству общих игр;
2. разнице в возрасте;
3. времени создания анкеты;
4. внутреннему id.

## История просмотра

Бот умеет возвращаться к уже просмотренным анкетам через таблицу `interactions`.

Для этого используются поля runtime-состояния:

- `browse_mode`;
- `history_candidate_action`;
- `history_cursor_id`;
- `history_cursor_created_at`.

Особенность:

- если анкета открыта из истории и ранее уже была лайкнута, изменить этот лайк уже нельзя;
- из истории можно вернуться к новым анкетам.

## Лайки, дизлайки и мэтчи

### Обычный лайк

Когда пользователь ставит лайк:

1. бот проверяет актуальность кандидата;
2. защищается от двойного нажатия;
3. сохраняет `like` в `interactions`;
4. проверяет, есть ли ответный лайк.

Если лайк взаимный:

- создается запись в `matches`;
- обе стороны получают уведомления о мэтче.

Если лайк не взаимный:

- создается или обновляется запись в `pending_likes`;
- при включенном `ENABLE_LIKE_NOTIFICATIONS` второй пользователь получает уведомление о входящем лайке.

### Лайк с сообщением

Пользователь может перейти в `STATE_LIKE_MESSAGE` и добавить текст до 500 символов.

После этого:

- сохраняется обычный `like` в `interactions`;
- текст сообщения сохраняется в `pending_likes.like_message`;
- при мэтче сообщение также может попасть в уведомление.

### Дизлайк

При дизлайке бот:

- сохраняет `dislike` в `interactions`;
- при необходимости закрывает соответствующий `pending_like`;
- сразу переходит к следующей анкете.

## Входящие лайки

Если у пользователя есть необработанный `pending_like`, бот может перевести его в `STATE_INCOMING_LIKE`.

Пользователь может:

- лайкнуть в ответ;
- отправить лайк с сообщением;
- поставить дизлайк;
- пожаловаться.

После ответа:

- запись в `pending_likes` помечается через `resolve_pending_like(...)`;
- бот либо показывает следующий входящий лайк, либо возвращает пользователя в предыдущий сценарий.

## Жалобы

Жалоба запускается для текущего активного профиля и переводит пользователя в `STATE_REPORT_REASON`.

Дальше бот:

1. ждет текст причины до 500 символов;
2. отправляет жалобу в чат модерации `peer_id = 2000000001`;
3. сохраняет `dislike` в `interactions`;
4. если был pending like, закрывает его как `dislike`.

В сообщение модерации входят:

- анкета пользователя, на которого жалуются;
- ссылка на диалог с жалующимся;
- ссылка на диалог с нарушителем;
- текст причины.

## Бан анкеты

У профиля есть поля:

- `is_banned`;
- `banned_at`;
- `ban_reason`.

Если пользователь забанен:

- бот не показывает его анкету другим;
- бот не дает ему полноценно пользоваться сценарием;
- при обращении в бот отправляет сообщение о блокировке.

Блокировка и разблокировка сейчас делаются отдельными SQL-скриптами:

- `sql/ban_profile.sql`
- `sql/unban_profile.sql`

## Ошибки доставки сообщений

Если VK не дает отправить пользователю сообщение, бот может пометить его профиль как недоступный для доставки.

Для этого используются поля:

- `delivery_disabled`;
- `delivery_error_code`;
- `delivery_error_at`.

Такие профили:

- исключаются из выдачи;
- не получают обычные уведомления.

Когда пользователь снова пишет боту, `reset_profile_delivery_status(...)` сбрасывает этот флаг.

## Деактивация анкеты

Пользователь может отключить анкету вручную:

1. бот переводит его в `STATE_DEACTIVATE_CONFIRM`;
2. ждет подтверждение;
3. сохраняет `is_active = 0`.

Отключенная анкета не участвует в выдаче.

Если затем пользователь снова идет смотреть анкеты, `reactivate_profile_if_needed(...)` автоматически возвращает `is_active = 1`.

## Полный сброс данных

Пользователь может полностью удалить свою анкету и связанную историю:

1. бот переводит его в `STATE_RESET_CONFIRM`;
2. после подтверждения вызывает `delete_user_data(...)`;
3. запись в `users` удаляется, а связанные данные уходят каскадно;
4. runtime-состояние очищается из памяти;
5. бот сразу запускает сценарий регистрации заново.

## Защита от повторных нажатий

Для ряда действий используется debounce-защита `is_duplicate_action(...)` с окном `0.5` секунды.

Она применяется, например, к:

- лайкам;
- дизлайкам;
- лайкам с сообщением;
- ответам на входящие лайки;
- завершению выбора игр;
- переходу к просмотру анкет.

Если действие признано дублем, бот игнорирует его и пишет это в лог действий.

## Логирование

В проекте есть два уровня логов:

- `logs/bot.log` — обычный технический лог;
- `logs/actions.jsonl` — структурированный журнал событий.

Логируются:

- входящие сообщения;
- исходящие сообщения;
- callback-события;
- ошибки отправки и редактирования сообщений;
- ошибки обработки событий;
- сохранение взаимодействий;
- итоговое состояние пользователя после сообщения;
- длительность обработки события.

## Сквозной сценарий работы

Упрощенно жизненный цикл выглядит так:

1. бот запускается и подключается к VK и MySQL;
2. проверяет runtime-схему БД;
3. начинает слушать Long Poll;
4. получает сообщение пользователя;
5. восстанавливает runtime-состояние;
6. при необходимости автозаполняет часть анкеты из VK;
7. ведет пользователя по регистрации или показывает уже заполненную анкету;
8. при переходе в просмотр подбирает подходящего кандидата;
9. сохраняет лайки, дизлайки, жалобы и мэтчи в БД;
10. после перезапуска продолжает сценарий из `user_sessions`.
