# Логика работы бота

GoParty реализован как VK-бот с серверной частью на FastAPI. Входящие события VK попадают в callback-обработчик, затем передаются во внутреннюю очередь и обрабатываются логикой из `bot_handlers`.

## Регистрация анкеты

Пользователь последовательно указывает:

- имя;
- возраст;
- пол;
- город;
- кого ищет;
- игры;
- наличие микрофона;
- описание;
- фотографии.

Основные поля анкеты сохраняются в `profiles`, выбранные игры - в `user_games`, фотографии - в `user_photos`, а настройки поиска - в `user_filters` и `user_filter_games`.

## Просмотр анкет

Подбор кандидата выполняется на основе:

- активности анкеты;
- отсутствия блокировки;
- заполненности обязательных полей;
- пола и возрастного диапазона из фильтров;
- совпадения игр;
- города;
- предпочтения по микрофону;
- истории предыдущих действий.

История действий хранится в `interactions`. Если пользователь уже поставил кандидату лайк или дизлайк, эта анкета не показывается повторно в обычном режиме просмотра.

## Лайки, дизлайки и мэтчи

Все действия пользователя по отношению к другим анкетам сохраняются в таблице `interactions`.

Обычный лайк записывается как:

```text
from_user_id -> to_user_id, action = like
```

Дизлайк записывается как:

```text
from_user_id -> to_user_id, action = dislike
```

Если пользователь отправляет лайк с сообщением, текст сохраняется в поле `interactions.like_message`.

Входящий лайк без ответа больше не хранится в отдельной таблице. Он определяется как лайк, направленный текущему пользователю, для которого еще нет встречного действия от текущего пользователя. Мэтч также не хранится отдельно: он определяется по двум встречным лайкам.

## Runtime-состояние

Состояние текущего диалога хранится в памяти приложения и дополнительно сохраняется в `user_sessions.session_json`. Это позволяет восстановить сценарий после перезапуска приложения или повторного обращения пользователя.

Ключевые runtime-поля:

- `step`
- `current_candidate`
- `browse_mode`
- `history_candidate_action`
- `history_cursor_id`
- `history_cursor_created_at`
- `pending_like_profile`
- `like_message_target_vk_user_id`
- `like_message_resume_step`
- `report_target_vk_user_id`
- `report_resume_step`
- `resume_step`

Постоянные пользовательские данные при этом остаются нормализованными в отдельных таблицах БД.
