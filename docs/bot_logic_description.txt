ЛОГИКА РАБОТЫ БОТА GO PARTY

1. Общая архитектура
Бот работает через VK Long Poll.
Главный цикл постоянно слушает события сообщества.
Поддерживаются 2 основных типа событий:
- MESSAGE_NEW: обычное новое сообщение пользователя.
- MESSAGE_EVENT: нажатие inline/callback-кнопок.

Для каждого события:
- фиксируется время начала обработки;
- определяется тип события;
- вызывается нужный обработчик;
- при ошибке она логируется;
- после завершения пишется лог длительности обработки.


2. Инициализация при запуске
При старте приложения:
- загружается токен VK и настройки БД из `config.py`;
- создаётся VK-сессия и объект API;
- создаётся объект Long Poll;
- открывается соединение с MySQL;
- в БД автоматически создаются/дополняются служебные таблицы и поля:
  pending_likes, user_sessions, а также runtime-поля профиля;
- запускается бесконечный цикл прослушивания событий.

Если Long Poll падает с ошибкой:
- ошибка логируется;
- бот ждёт 3 секунды;
- создаёт Long Poll заново;
- продолжает работу.


3. Где хранится состояние
Состояние пользователя хранится в двух местах:
- в памяти процесса, в словаре `users`;
- в БД, в таблице `user_sessions`.

Постоянные данные анкеты хранятся в БД:
- users;
- profiles;
- user_games;
- user_photos;
- interactions;
- matches;
- pending_likes.

В runtime-состоянии хранятся:
- текущий шаг диалога `step`;
- текущая показываемая анкета `current_candidate`;
- режим просмотра `browse_mode` (новые анкеты или история);
- служебные курсоры истории;
- данные о входящем лайке;
- данные о жалобе;
- данные о сообщении к лайку;
- признаки редактирования фото;
- шаг, на который нужно вернуться после временного сценария.

После каждого сообщения и callback-события runtime-состояние сохраняется в `user_sessions`.


4. Основные состояния диалога
Бот использует state-machine. Основные состояния:
- REG_NAME: ввод имени;
- REG_AGE: ввод возраста;
- REG_GENDER: выбор пола;
- REG_CITY: ввод города;
- REG_LOOKING: выбор, кого ищет пользователь;
- STATE_GAMES: выбор игр через inline-кнопки;
- STATE_ABOUT: ввод текста о себе;
- STATE_PHOTO: загрузка фото;
- STATE_PHOTO_MORE: предложение добавить ещё фото;
- STATE_REVIEW: показ своей анкеты и меню управления;
- STATE_EDIT_MENU: меню редактирования основных полей;
- STATE_BROWSE: просмотр других анкет;
- STATE_INCOMING_LIKE: обработка входящего лайка;
- STATE_LIKE_MESSAGE: ввод сообщения вместе с лайком;
- STATE_REPORT_REASON: ввод причины жалобы;
- STATE_DEACTIVATE_CONFIRM: подтверждение отключения анкеты;
- STATE_RESET_CONFIRM: подтверждение полного сброса данных.


5. Что происходит при новом сообщении
Когда приходит MESSAGE_NEW:
- логируется входящее сообщение;
- если у пользователя ранее была пометка, что ему нельзя доставлять сообщения, она сбрасывается;
- создаётся локальная функция `send(...)` для отправки сообщений и логирования;
- текст нормализуется;
- payload, если есть, парсится.

Дальше логика такая:
- если пользователь есть в памяти, но его профиль удалён из БД, он убирается из `users`;
- если пользователя ещё нет в runtime, запускается стартовый сценарий `start_bot_flow`;
- если пользователь написал "Начать" или пришёл payload-команда `start`, тоже запускается `start_bot_flow`;
- иначе бот смотрит на текущее состояние `step` и передаёт управление нужному обработчику.


6. Стартовый сценарий `start_bot_flow`
При старте работы с пользователем:
- проверяется, существовал ли его профиль в БД раньше;
- создаётся или загружается пользователь;
- подгружаются данные анкеты из БД;
- подгружается сохранённое runtime-состояние;
- запрашиваются данные VK-профиля пользователя (имя, возраст, город, пол), если доступны;
- пустые поля анкеты могут автоматически заполниться данными из VK.

Дальше:
- если профиль существовал раньше, бот пишет приветствие о возвращении;
- если анкета уже полная:
  - если есть входящие лайки, показывается первый из них;
  - иначе показывается экран своей анкеты (`STATE_REVIEW`);
- если анкета неполная:
  - бот последовательно переводит пользователя на первый незаполненный обязательный шаг.

Порядок обязательных шагов:
1. Имя
2. Возраст
3. Пол
4. Город
5. Кого ищет
6. Игры
7. Описание
8. Фото

Если всё заполнено:
- показывается экран обзора своей анкеты.


7. Сценарий регистрации и заполнения анкеты
7.1. Имя
- пользователь вводит имя;
- если имя пустое или длиннее 50 символов, бот просит ввести корректно;
- если всё корректно, поле сохраняется в БД;
- далее бот переводит пользователя к следующему незаполненному шагу.

7.2. Возраст
- принимается только число;
- возраст должен быть от 14 до 99;
- иначе бот просит ввести корректное значение;
- при успехе возраст сохраняется.

7.3. Пол
- доступны варианты: парень или девушка;
- сохраняется код `male` или `female`;
- иначе бот просит нажать кнопку.

7.4. Город
- принимается непустой текст до 50 символов;
- при успехе сохраняется.

7.5. Кого ищет
- варианты: парня, девушку, без разницы;
- сохраняется `male`, `female` или `any`.

7.6. Игры
- это отдельный callback-сценарий через inline-кнопки;
- пользователь включает/выключает игры;
- после нажатия "Готово":
  - выбор сохраняется в БД;
  - бот убирает inline-клавиатуру у сообщения;
  - переходит к следующему незаполненному шагу.

7.7. Описание
- принимается только текст;
- вложения не допускаются;
- максимум 2000 символов;
- при успехе поле сохраняется.

7.8. Фото
- принимаются только фото;
- можно сохранить от 1 до 3 фото;
- если фотографий меньше 3, бот предлагает добавить ещё;
- если фотографий стало 3, анкета считается готовой по фото;
- если попытаться прислать больше 3 фото, бот пишет ошибку.

После заполнения всех обязательных полей:
- пользователь попадает на экран своей анкеты (`STATE_REVIEW`).


8. Экран своей анкеты `STATE_REVIEW`
На этом экране бот показывает:
- текст анкеты;
- прикреплённые фото;
- клавиатуру действий.

Доступные действия:
- "Да, смотреть анкеты";
- "Изменить информацию о себе";
- "Изменить описание анкеты";
- "Изменить игры";
- "Изменить фото";
- "Сбросить данные";
- "Отключить анкету".

Логика:
- "Да, смотреть анкеты" -> переход к просмотру анкет;
- "Изменить информацию о себе" -> `STATE_EDIT_MENU`;
- "Изменить описание анкеты" -> `STATE_ABOUT`;
- "Изменить игры" -> `STATE_GAMES`;
- "Изменить фото" -> переход в `STATE_PHOTO`;
- "Сбросить данные" -> `STATE_RESET_CONFIRM`;
- "Отключить анкету" -> `STATE_DEACTIVATE_CONFIRM`.


9. Редактирование анкеты
9.1. Меню редактирования `STATE_EDIT_MENU`
Пользователь выбирает, что менять:
- имя;
- возраст;
- пол;
- кого ищет;
- город;
- назад.

После выбора бот переводит в соответствующий state ввода поля.
После сохранения обновлённого поля:
- бот пытается провести пользователя по следующему обязательному шагу, если что-то стало пустым;
- либо возвращает к экрану анкеты.

9.2. Изменение описания
- переводит в `STATE_ABOUT`;
- принимает только текст;
- после сохранения возвращает к анкете.

9.3. Изменение игр
- запускает заново `STATE_GAMES`;
- после "Готово" сохраняет выбор и возвращает к анкете.

9.4. Изменение фото
- если фото уже были, бот предлагает либо оставить как есть, либо прислать новые;
- после успешного сохранения фото, если анкета полная, `is_active` снова ставится в 1;
- затем показывается обновлённая анкета.


10. Как бот выбирает кандидатов для показа
При показе новой анкеты используется запрос `get_random_candidate`.
Кандидат должен удовлетворять условиям:
- это не сам текущий пользователь;
- анкета активна;
- анкета не забанена;
- у анкеты не отключена доставка сообщений;
- заполнены основные поля анкеты;
- текущий пользователь ещё не ставил этой анкете like/dislike;
- кандидат раньше не ставил текущему пользователю dislike.

Дополнительно учитывается совместимость:
- если пользователь ищет конкретный пол, кандидат должен ему соответствовать;
- если у пользователя указан свой пол, кандидат должен искать либо этот пол, либо "any".

Приоритет показа:
- сначала выше анкеты с большим количеством общих игр;
- внутри одинакового приоритета выбор случайный.


11. Просмотр анкет `STATE_BROWSE`
В этом состоянии возможны 2 режима:
- `browse_mode = new`: просмотр новых анкет;
- `browse_mode = history`: возврат к уже просмотренным анкетам по истории действий.

Если вызывается показ новой анкеты:
- профиль пользователя при необходимости реактивируется (`is_active = 1`);
- выбирается случайный кандидат;
- его `vk_user_id` записывается в `current_candidate`;
- пользователю отправляется анкета и клавиатура действий.

Если кандидата нет:
- бот пишет, что анкет пока нет;
- показывает клавиатуру с переходом к своей анкете или повторному поиску.

Действия в `STATE_BROWSE`:
- ❤️ -> лайк;
- 💌 -> лайк с сообщением;
- 👎 -> дизлайк;
- "Вернуться к предыдущей анкете" -> открыть прошлую анкету из истории;
- "Вернуться к новым анкетам" -> выйти из режима history и продолжить обычный просмотр;
- "Моя анкета" -> вернуться к экрану своей анкеты;
- "Пожаловаться" -> открыть сценарий жалобы.


12. История просмотренных анкет
При нажатии "Вернуться к предыдущей анкете":
- бот ищет последнюю запись в `interactions` от этого пользователя;
- если просмотр уже идёт в history-режиме, бот берёт запись ещё раньше текущего курсора;
- если запись найдена, в `current_candidate` загружается её профиль;
- `history_candidate_action` хранит, был там like или dislike;
- пользователь видит старую анкету.

Если старых анкет больше нет:
- бот сообщает об этом;
- предлагает вернуться к новым анкетам.

Ограничение:
- если в history-режиме у этой анкеты уже был like, изменить оценку нельзя;
- можно только пожаловаться, смотреть ещё более старые или вернуться к новым.


13. Обычный лайк
Когда пользователь нажимает ❤️ в просмотре анкеты:
- проверяется защита от двойного нажатия за короткое время;
- проверяется, что кандидат существует, активен, не равен самому пользователю и доступен для доставки;
- действие сохраняется в `interactions` как `like`;
- если на этого пользователя раньше был входящий pending like, он помечается как обработанный.

После этого 2 варианта:
- если у второго пользователя уже был like в ответ, возникает мэтч;
- если ответного like ещё не было, создаётся запись в `pending_likes`.

Если создан новый `pending_likes` и включены уведомления:
- целевому пользователю сразу отправляется уведомление, что его анкета кому-то понравилась.

После обычного лайка:
- текущий кандидат сбрасывается;
- если пользователь был в history-режиме, бот возвращает его в режим новых анкет;
- показывается следующая анкета.


14. Лайк с сообщением
Когда пользователь нажимает 💌:
- бот определяет целевой профиль;
- сохраняет, на какой экран потом нужно вернуться;
- переводит пользователя в `STATE_LIKE_MESSAGE`;
- просит ввести текст до 500 символов.

Дальше:
- если пользователь нажал "Вернуться назад", бот возвращается на предыдущий экран;
- если прислан невалидный ввод, бот просит только текст до 500 символов;
- если целевая анкета уже недоступна, бот выходит обратно в просмотр.

Если текст валиден:
- в `interactions` сохраняется like;
- в `pending_likes` сохраняется pending like вместе с текстом сообщения;
- если уже есть ответный лайк, сразу создаётся мэтч;
- если мэтча нет, второй стороне может уйти уведомление о лайке с сообщением.

После завершения:
- бот возвращает пользователя либо на экран входящего лайка, либо к просмотру анкет, либо к своей анкете, в зависимости от того, откуда был начат сценарий.


15. Входящий лайк `STATE_INCOMING_LIKE`
Этот сценарий запускается, если:
- у пользователя есть необработанный pending like;
- бот обнаружил его при старте;
- или пользователю только что отправили live-уведомление о лайке.

Перед входом в сценарий:
- бот запоминает, на каком шаге пользователь был до этого (`resume_step`);
- в `pending_like_profile` записывает профиль того, кто лайкнул.

Пользователь видит:
- сообщение, что его анкета кому-то понравилась;
- анкету лайкнувшего;
- возможное прикреплённое текстовое сообщение;
- кнопки: ❤️, 💌, 👎, пожаловаться.

Варианты действий:
- ❤️:
  - сохраняется like в ответ;
  - pending like закрывается;
  - если это взаимный лайк, обоим пользователям отправляется сообщение о мэтче;
  - затем бот либо показывает следующий pending like, либо возвращает пользователя к предыдущему сценарию.
- 💌:
  - запускается сценарий `STATE_LIKE_MESSAGE`;
  - после завершения бот возвращает пользователя обратно к входящему лайку или дальше по очереди.
- 👎:
  - сохраняется dislike;
  - pending like закрывается;
  - далее показывается следующий pending like либо выполняется возврат назад.
- "Пожаловаться":
  - запускается сценарий жалобы именно на этого пользователя.


16. Мэтч
Мэтч возникает, если:
- пользователь поставил like кандидату;
- и у кандидата уже есть like в ответ в таблице `interactions`.

Когда мэтч найден:
- в таблицу `matches` добавляется пара пользователей;
- обоим отправляется сообщение о мэтче;
- в сообщении даётся ссылка на страницу второго пользователя `https://vk.com/id...`;
- если мэтч возник после лайка с сообщением, сообщение тоже добавляется в уведомление для второй стороны.


17. Жалоба `STATE_REPORT_REASON`
Сценарий жалобы можно запустить:
- из режима просмотра анкеты;
- из режима входящего лайка.

При старте сценария:
- бот сохраняет, на кого жалуются;
- запоминает, с какого шага был вызван сценарий;
- переводит пользователя в `STATE_REPORT_REASON`;
- просит описать причину.

Дальше:
- если пользователь отменяет жалобу, бот возвращает его на предыдущий экран;
- если причина пустая, слишком длинная или пришла с вложениями, бот просит только текст до 500 символов;
- если анкета уже недоступна, бот переходит к следующей анкете.

Если причина валидна:
- бот отправляет сообщение в чат модерации `peer_id = 2000000001`;
- туда включается анкета нарушителя, ссылка на диалог жалующегося и ссылка на диалог нарушителя;
- одновременно в `interactions` сохраняется `dislike`;
- если был pending like, он помечается как обработанный dislike;
- затем бот сообщает, что жалоба отправлена, и показывает следующую анкету.


18. Обычный дизлайк
Когда пользователь нажимает 👎 при просмотре анкеты:
- проверяется защита от дубля;
- проверяется актуальность кандидата;
- в `interactions` сохраняется `dislike`;
- pending like при необходимости закрывается как `dislike`;
- текущий кандидат сбрасывается;
- если просмотр был в history-режиме, бот возвращается в обычный режим новых анкет;
- показывается следующая анкета.


19. Отключение анкеты
Если пользователь нажал "Отключить анкету":
- бот переводит его в `STATE_DEACTIVATE_CONFIRM`;
- просит подтвердить действие.

Если пользователь подтверждает:
- в профиле сохраняется `is_active = 0`;
- текущий кандидат очищается;
- бот пишет подтверждение и оставляет клавиатуру возврата.

Если пользователь нажимает "Назад":
- бот возвращает его к экрану своей анкеты.

Даже если анкета отключена:
- пользователь может снова нажать "Смотреть анкеты";
- тогда при показе новых анкет бот автоматически реактивирует профиль (`is_active = 1`).


20. Полный сброс данных
Если пользователь нажимает "Сбросить данные":
- бот переводит его в `STATE_RESET_CONFIRM`;
- просит подтверждение полного удаления.

Если пользователь подтверждает:
- удаляется запись пользователя из таблицы `users`;
- каскадно удаляются связанные данные анкеты и истории;
- runtime-пользователь удаляется из словаря `users`;
- бот сообщает об очистке;
- сразу запускает регистрацию заново через `start_bot_flow`.

Если пользователь нажимает "Назад":
- возврат на экран своей анкеты.


21. Callback-логика выбора игр `MESSAGE_EVENT`
Если приходит callback-событие и пользователь находится в `STATE_GAMES`:
- `toggle_game`:
  - бот включает или выключает флаг выбранной игры;
  - редактирует текущее сообщение, обновляя галочки на inline-кнопках;
  - отправляет подтверждение события VK.
- `games_done`:
  - бот защищается от двойного нажатия;
  - фиксирует, что шаг игр завершён;
  - сохраняет список игр в БД;
  - заменяет inline-сообщение на краткое текстовое резюме выбранных игр;
  - затем переводит пользователя либо на следующий обязательный шаг, либо на экран своей анкеты.

Если callback не относится к ожидаемому сценарию:
- бот просто отвечает VK, чтобы событие считалось обработанным.


22. Автовосстановление пользователя
Если пользователь уже есть в БД:
- бот может восстановить его анкету и runtime-состояние после перезапуска процесса;
- для этого при входе вызывается `ensure_runtime_user`;
- runtime-состояние собирается из:
  - данных профиля из БД;
  - `user_sessions` из БД;
  - уже существующего объекта в памяти;
  - данных VK-профиля.

Это позволяет:
- не терять текущий шаг диалога после перезапуска;
- не терять pending like;
- не терять текущую анкету просмотра;
- продолжать сценарий с того же места.


23. Логика доставки сообщений и недоступных пользователей
Все отправки идут через safe-обёртки:
- `safe_vk_send`;
- `safe_vk_edit`;
- `safe_vk_answer_event`.

Если отправка сообщения в личку не удалась по причинам вроде:
- пользователь удалён;
- переписка запрещена;
- ограничения приватности;
- нет разрешения на сообщения,
то профиль пользователя помечается как:
- `delivery_disabled = 1`;
- сохраняется код ошибки и время.

Такие профили:
- не должны выбираться как кандидаты для показа;
- не должны получать обычные уведомления.

Если пользователь потом снова написал боту:
- в начале обработки входящего сообщения эта блокировка доставки сбрасывается.


24. Побочные эффекты при изменении анкеты
Если пользователь меняет важные поля анкеты:
- имя;
- возраст;
- город;
- описание;
- пол;
- игры;
- фото,
то бот очищает полученные ранее дизлайки в адрес этого пользователя.

Смысл:
- после заметного изменения анкеты пользователя можно снова показывать тем, кто раньше его дизлайкнул.


25. Защита от повторных нажатий
Для некоторых действий используется debounce около 0.5 секунды.
Если пользователь слишком быстро нажимает одну и ту же кнопку повторно:
- действие игнорируется;
- это логируется как `duplicate_action_ignored`.

Защита применяется, например, для:
- лайков;
- дизлайков;
- лайка с сообщением;
- подтверждения выбора игр;
- перехода к просмотру анкет.


26. Краткая сквозная последовательность работы бота
Ниже упрощённый линейный сценарий для диаграммы высокого уровня:

1. Бот запускается.
2. Подключается к VK API и БД.
3. Подготавливает таблицы и поля в БД.
4. Начинает слушать Long Poll.
5. Получает событие.
6. Если это новое сообщение:
- восстанавливает/создаёт runtime пользователя;
- определяет текущее состояние;
- передаёт сообщение в обработчик состояния.
7. Если это callback:
- обрабатывает inline-логику выбора игр.
8. Если анкета неполная:
- бот пошагово собирает обязательные поля.
9. Когда анкета заполнена:
- показывает экран своей анкеты.
10. Пользователь переходит к просмотру других анкет.
11. Бот подбирает подходящего кандидата из БД.
12. Пользователь ставит like / like с сообщением / dislike / жалобу / открывает историю.
13. Бот сохраняет действие в БД.
14. Если лайк взаимный:
- создаёт мэтч и уведомляет обе стороны.
15. Если лайк не взаимный:
- создаёт pending like и при необходимости отправляет уведомление второй стороне.
16. Если на пользователя кто-то лайкнул:
- бот показывает сценарий входящего лайка.
17. Пользователь может редактировать анкету, отключить её или полностью сбросить данные.
18. После каждого шага состояние сохраняется, чтобы продолжить работу после перезапуска.


27. Что удобно рисовать на диаграмме отдельно
Для наглядной диаграммы лучше сделать отдельные блок-схемы:
- Главный цикл обработки событий;
- Регистрация анкеты;
- Просмотр анкет и выбор кандидата;
- Логика лайка, pending like и мэтча;
- Сценарий входящего лайка;
- Сценарий жалобы;
- Редактирование / отключение / сброс анкеты;
- Восстановление состояния пользователя после перезапуска.
