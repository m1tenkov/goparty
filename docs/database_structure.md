# Структура базы данных `goparty_bot`

Этот документ описывает назначение базы данных проекта `GoParty`, состав таблиц, их поля, связи и роль каждой сущности в работе VK-бота.

## Общее назначение базы данных

База данных `goparty_bot` используется для хранения:

- зарегистрированных пользователей бота;
- анкет пользователей;
- выбранных игровых интересов;
- фотографий пользователей;
- истории лайков и дизлайков;
- входящих лайков, ожидающих ответа;
- взаимных совпадений;
- текущего состояния диалога пользователя с ботом.

Центральной таблицей является `users`. С ней связаны все остальные сущности системы.

## Общая логика структуры

Работа базы строится по следующему принципу:

1. Пользователь начинает взаимодействие с ботом и появляется в таблице `users`.
2. Основные данные анкеты хранятся в таблице `profiles`.
3. Выбранные игры сохраняются через связку `games` и `user_games`.
4. Фотографии пользователя сохраняются в `user_photos`.
5. Лайки и дизлайки фиксируются в `interactions`.
6. Входящие лайки, на которые еще не ответили, сохраняются в `pending_likes`.
7. Взаимные симпатии фиксируются в `matches`.
8. Текущее состояние сценария диалога хранится в `user_sessions`.

##############
# Таблица `users`
##############

Основная таблица пользователей системы.

### Назначение

Хранит список всех пользователей, которые хотя бы один раз взаимодействовали с ботом.

### Поля

- `id` — внутренний идентификатор пользователя в базе данных. Используется как основной ключ и как ссылка из других таблиц.
- `vk_user_id` — идентификатор пользователя ВКонтакте. По нему бот связывает запись в базе с реальным пользователем VK.
- `created_at` — дата и время первого добавления пользователя в систему.

### Особенности

- `id` является первичным ключом.
- `vk_user_id` имеет уникальное ограничение, чтобы один и тот же пользователь VK не был добавлен в базу дважды.

##############
# Таблица `profiles`
##############

Таблица основной анкеты пользователя.

### Назначение

Содержит персональные данные анкеты, а также служебные признаки, влияющие на показ анкеты, модерацию и доставку сообщений.

### Поля

- `user_id` — ссылка на таблицу `users`. Одновременно является первичным ключом таблицы, то есть одному пользователю соответствует одна анкета.
- `name` — имя пользователя, отображаемое в анкете.
- `age` — возраст пользователя.
- `city` — город пользователя.
- `about` — текстовое описание пользователя.
- `gender` — пол пользователя (`male` или `female`).
- `looking_for` — кого пользователь ищет (`male`, `female`, `any`).
- `is_active` — активна ли анкета. Если значение равно `0`, анкета не участвует в показе другим пользователям.
- `updated_at` — дата и время последнего изменения анкеты.
- `is_banned` — признак блокировки анкеты.
- `banned_at` — дата и время блокировки.
- `ban_reason` — причина блокировки анкеты.
- `delivery_disabled` — признак того, что пользователю нельзя доставить сообщение.
- `delivery_error_code` — код ошибки доставки сообщения от VK API.
- `delivery_error_at` — время последней ошибки доставки.
- `games_step_completed` — служебный признак того, что пользователь завершил шаг выбора игр.

### Особенности

- связь с `users` реализована по `user_id`;
- при удалении пользователя его анкета удаляется автоматически за счет `ON DELETE CASCADE`.

##############
# Таблица `games`
##############

Справочник доступных игр.

### Назначение

Содержит фиксированный набор игр, которые пользователь может выбрать при заполнении анкеты.

### Поля

- `id` — внутренний идентификатор игры.
- `code` — короткий системный код игры, используемый в коде и обработчиках.
- `title` — отображаемое название игры.

### Особенности

- `code` уникален;
- таблица используется как справочник и не зависит от конкретного пользователя.

##############
# Таблица `user_games`
##############

Связующая таблица между пользователями и играми.

### Назначение

Хранит информацию о том, какие игры выбрал каждый пользователь.

### Поля

- `user_id` — ссылка на пользователя из таблицы `users`.
- `game_id` — ссылка на игру из таблицы `games`.
- `created_at` — время добавления игры пользователю.

### Особенности

- первичный ключ составной: (`user_id`, `game_id`);
- это исключает дублирование одной и той же игры у одного пользователя;
- таблица реализует связь многие-ко-многим между `users` и `games`.

##############
# Таблица `user_photos`
##############

Таблица фотографий пользователя.

### Назначение

Содержит фотографии, которые пользователь прикрепил к своей анкете.

### Поля

- `id` — идентификатор записи фотографии.
- `user_id` — ссылка на пользователя из таблицы `users`.
- `photo_token` — идентификатор фотографии во ВКонтакте, используемый для прикрепления изображения к сообщениям.
- `sort_order` — порядковый номер фотографии в анкете.
- `created_at` — время добавления фотографии.

### Особенности

- один пользователь может иметь несколько фотографий;
- ограничение `uq_user_photos_user_sort` не позволяет сохранить две фотографии с одинаковым порядком у одного пользователя;
- при удалении пользователя его фотографии удаляются автоматически.

##############
# Таблица `interactions`
##############

Таблица пользовательских взаимодействий.

### Назначение

Хранит основные действия пользователя по отношению к другим пользователям: лайк или дизлайк.

### Поля

- `id` — идентификатор взаимодействия.
- `from_user_id` — пользователь, который совершил действие.
- `to_user_id` — пользователь, в отношении которого совершено действие.
- `action` — тип действия: `like` или `dislike`.
- `created_at` — дата и время фиксации действия.

### Особенности

- ограничение `uq_interaction_pair` не позволяет одному пользователю многократно сохранять итоговое действие по отношению к одному и тому же другому пользователю;
- индекс `idx_interactions_to_user` ускоряет поиск действий, направленных на конкретного пользователя.

##############
# Таблица `pending_likes`
##############

Таблица входящих лайков, ожидающих ответа.

### Назначение

Используется для хранения лайков, которые один пользователь отправил другому, но второй пользователь еще не дал ответ.

### Поля

- `id` — идентификатор записи.
- `liker_user_id` — пользователь, поставивший лайк.
- `target_user_id` — пользователь, получивший лайк.
- `response_action` — ответ целевого пользователя (`like` или `dislike`). До ответа имеет значение `NULL`.
- `notified_at` — время, когда пользователю было отправлено уведомление о входящем лайке.
- `responded_at` — время, когда пользователь ответил на лайк.
- `created_at` — время создания записи о лайке.
- `like_message` — текст, прикрепленный к лайку, если он был отправлен с сообщением.

### Особенности

- ограничение `uq_pending_like_pair` исключает дублирование ожидающего лайка между одной и той же парой пользователей;
- индекс `idx_pending_likes_target_pending` помогает быстро находить необработанные входящие лайки конкретного пользователя.

##############
# Таблица `matches`
##############

Таблица взаимных совпадений.

### Назначение

Фиксирует ситуацию, когда два пользователя поставили друг другу лайк и между ними образовалось совпадение.

### Поля

- `id` — идентификатор совпадения.
- `user1_id` — первый пользователь в паре совпадения.
- `user2_id` — второй пользователь в паре совпадения.
- `created_at` — время создания совпадения.

### Особенности

- ограничение `uq_match_pair` не позволяет создать одно и то же совпадение повторно;
- таблица хранит только факт взаимного совпадения и время его появления.

##############
# Таблица `user_sessions`
##############

Таблица пользовательских сессий.

### Назначение

Сохраняет текущее состояние диалога пользователя с ботом, чтобы после повторного обращения или перезапуска приложения можно было продолжить сценарий с нужного места.

### Поля

- `user_id` — ссылка на пользователя из таблицы `users`.
- `session_json` — JSON-структура с текущим состоянием пользователя.
- `updated_at` — дата и время последнего обновления состояния.

### Что хранится в `session_json`

В поле `session_json` могут храниться:

- текущий шаг диалога (`step`);
- текущая просматриваемая анкета (`current_candidate`);
- режим просмотра (`browse_mode`);
- признаки завершения отдельных шагов;
- временные служебные поля для сценариев лайка с сообщением;
- временные данные для жалобы;
- данные для возврата к предыдущему состоянию;
- информация о входящем лайке.

### Особенности

- таблица особенно важна для состоянийного бота;
- без нее после перезапуска приложения пользовательский диалог пришлось бы начинать заново.

## Связи между таблицами

Основные связи в базе данных:

- `users` → `profiles` — один к одному;
- `users` → `user_photos` — один ко многим;
- `users` → `user_sessions` — один к одному;
- `users` ↔ `games` через `user_games` — многие ко многим;
- `users` → `interactions` — один ко многим в двух ролях: отправитель и получатель действия;
- `users` → `pending_likes` — один ко многим в двух ролях: отправитель лайка и получатель лайка;
- `users` → `matches` — один ко многим в двух ролях: первый и второй участник совпадения.

## Почему структура базы данных удобна

Такая структура позволяет:

- разделять основные данные анкеты и служебную информацию;
- не дублировать данные об играх;
- хранить несколько фотографий на одного пользователя;
- отдельно учитывать действия пользователей;
- удобно обрабатывать входящие лайки и взаимные совпадения;
- восстанавливать пользовательское состояние после перезапуска бота;
- поддерживать целостность данных за счет внешних ключей и каскадного удаления.

## Итог

База данных `goparty_bot` построена по реляционному принципу и охватывает все ключевые аспекты работы бота: регистрацию пользователей, хранение анкет, выбор интересов, загрузку фотографий, обработку взаимодействий и сохранение состояния пользовательского диалога. Такая структура делает систему устойчивой, расширяемой и удобной для сопровождения.
