# `user_sessions.session_json`

Этот документ поясняет, зачем в проекте нужна таблица `user_sessions` и что именно хранится в поле `session_json`.

## Зачем это нужно

Бот работает как stateful-сценарий: пользователь проходит регистрацию, редактирование анкеты, просмотр анкет, обработку входящих лайков и жалоб. Чтобы после следующего сообщения или после перезапуска приложения не терять контекст, часть runtime-состояния сохраняется в базе.

Именно для этого используется таблица `user_sessions`:

- `user_id` связывает запись с пользователем из таблицы `users`;
- `session_json` хранит сериализованное состояние диалога;
- `updated_at` показывает, когда состояние обновлялось последний раз.

Если состояние пустое, запись для пользователя удаляется. Если состояние есть, оно сохраняется как JSON строкой через `INSERT ... ON DUPLICATE KEY UPDATE`.

Основная логика находится в:

- [database.py](/abs/path/c:/Projects/VS%20Code/database.py:76) - создание таблицы;
- [database.py](/abs/path/c:/Projects/VS%20Code/database.py:501) - загрузка `session_json`;
- [database.py](/abs/path/c:/Projects/VS%20Code/database.py:527) - сохранение `session_json`;
- [bot_handlers/utils.py](/abs/path/c:/Projects/VS%20Code/bot_handlers/utils.py:202) - список runtime-полей, которые реально пишутся в БД.

## Что хранится в `session_json`

Поле не хранит полную анкету пользователя. Анкета, игры, фото и постоянные данные лежат в других таблицах. В `session_json` кладется именно временный контекст, нужный для продолжения сценария.

На текущий момент в проекте сохраняются такие ключи:

- `step` - текущий шаг диалога, например регистрация, просмотр анкет, ввод сообщения к лайку.
- `current_candidate` - `vk_user_id` анкеты, которую пользователь сейчас смотрит.
- `games_step_completed` - завершал ли пользователь шаг выбора игр.
- `browse_mode` - режим просмотра анкет, например новые анкеты или история.
- `history_candidate_action` - какое действие было связано с анкетой в истории.
- `history_cursor_id` - служебный курсор для пагинации истории.
- `history_cursor_created_at` - временная метка курсора истории.
- `pending_like_profile` - данные о входящем лайке, который сейчас ожидает обработки.
- `like_message_target_vk_user_id` - кому пользователь сейчас пишет сообщение вместе с лайком.
- `like_message_resume_step` - на какой шаг надо вернуться после отправки или отмены сообщения к лайку.
- `report_target_vk_user_id` - пользователь, на которого сейчас оформляется жалоба.
- `report_resume_step` - на какой шаг надо вернуться после жалобы.
- `had_photos_before_edit` - были ли фото до начала их редактирования.
- `resume_step` - общий служебный шаг возврата после промежуточного сценария.
- `suppress_post_match_prompt` - флаг подавления дополнительного prompt после мэтча.

## Что это означает на практике

Примерно так может выглядеть содержимое `session_json`:

```json
{
  "step": "STATE_BROWSE",
  "current_candidate": 123456789,
  "games_step_completed": true,
  "browse_mode": "new",
  "history_candidate_action": null,
  "history_cursor_id": null,
  "history_cursor_created_at": null,
  "pending_like_profile": null,
  "like_message_target_vk_user_id": null,
  "like_message_resume_step": null,
  "report_target_vk_user_id": null,
  "report_resume_step": null,
  "had_photos_before_edit": false,
  "resume_step": null,
  "suppress_post_match_prompt": false
}
```

Это означает не "данные анкеты", а "где пользователь сейчас находится внутри логики бота и что нужно восстановить при следующем сообщении".

## Чего там нет

В `session_json` не должны храниться постоянные пользовательские данные, для которых уже есть нормализованные таблицы:

- имя, возраст, город, `about`;
- пол и кого ищет пользователь;
- игры пользователя;
- фотографии;
- история лайков, мэтчей, жалоб и других постоянных сущностей.

Такие данные берутся из таблиц `profiles`, `profile_games`, `profile_photos`, `likes`, `matches`, `complaints` и других, а `session_json` только помогает продолжить незавершенный сценарий.

## Итог

`user_sessions.session_json` - это слой восстановления контекста. Он нужен, чтобы бот:

- не забывал текущий шаг пользователя;
- умел продолжить сценарий после рестарта;
- корректно возвращал пользователя из промежуточных flow, например после лайка с сообщением или жалобы;
- не смешивал временное runtime-состояние с постоянными данными анкеты.
