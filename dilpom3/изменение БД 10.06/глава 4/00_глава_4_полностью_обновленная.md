# ГЛАВА 4. РАЗРАБОТКА И РЕАЛИЗАЦИЯ ПРОГРАММНОГО КОДА И УДАЛЕННОГО СЕРВЕРА

# 4.1. Реализация серверной части ВК-бота

Серверная часть проекта ГоПати реализована на языке Python и представляет собой приложение, обеспечивающее прием событий от платформы «ВКонтакте», маршрутизацию пользовательских действий, взаимодействие с базой данных и отправку ответных сообщений [14]. На уровне архитектуры решение разделено на несколько логических компонентов: HTTP-точка входа для Callback API, обработчик событий, модуль работы с VK API, слой бизнес-логики диалога и слой доступа к данным [5; 7; 17; 18].

В качестве внешней точки входа используется приложение «FastAPI» [19], реализованное в файле «app.py». Данный компонент принимает HTTP-запросы от платформы «ВКонтакте» по маршруту «/vk/callback», проверяет корректность секретного ключа «VK_CALLBACK_SECRET», обрабатывает событие подтверждения сервера через «confirmation token», а для рабочих событий помещает полезную нагрузку во внутреннюю очередь обработки. Дополнительно в приложении реализован технический endpoint «/health», позволяющий выполнять проверку доступности сервиса после развертывания.

В файле «app.py» создается объект приложения FastAPI, регистрируются маршруты «/health» и «/vk/callback», выполняется проверка секретного ключа callback-запроса, обрабатывается событие подтверждения сервера, а также настраиваются события жизненного цикла приложения: при запуске стартует worker для обработки входящих событий, а при завершении работы приложения он останавливается.

Такая реализация позволяет быстро подтвердить получение события со стороны VK и не выполнять длительную пользовательскую логику непосредственно внутри HTTP-запроса. Если событие относится к типам «message_new» или «message_event», оно помещается во внутреннюю очередь. Если очередь заполнена, приложение возвращает ошибку 503, что позволяет явно зафиксировать ситуацию перегрузки.

Обработка входящих событий вынесена в модуль «event_processing.py». В рамках данного слоя различаются два основных типа событий: «message_new», соответствующий обычным входящим сообщениям пользователя, и «message_event», соответствующий callback-событиям от inline-кнопок. Также модуль содержит внутреннюю очередь callback-событий и worker-поток, который запускается при старте приложения, забирает события из очереди и передает их в прикладные обработчики. Такой подход позволяет FastAPI быстро вернуть платформе «ВКонтакте» ответ «ok», а пользовательский сценарий обработать отдельно, не задерживая callback-запрос.

Для взаимодействия с API социальной сети используется модуль «vk_bot.py», создающий объект VK API на основе токена сообщества [16; 17]. Через данный объект выполняется отправка сообщений пользователям, ответ на callback-события, получение открытых данных профиля и другие сетевые вызовы, необходимые для функционирования бота.

Центральный прикладной компонент серверной части размещен в модуле «bot_handlers/router.py». Именно он реализует основную диалоговую логику: стартовое взаимодействие, последовательное заполнение анкеты, показ собственной анкеты, просмотр анкет других пользователей, работу с лайками, жалобами, мэтчами, редактированием профиля и восстановлением состояния после повторного обращения пользователя.

Таким образом, серверная часть ГоПати построена по модульному принципу и сочетает в себе HTTP-интерфейс на базе «FastAPI», интеграцию с «VK API», очередь обработки callback-событий, прикладную обработку пользовательских сценариев и отдельный слой работы с данными. Такая архитектура обеспечивает разделение ответственности между компонентами, упрощает сопровождение проекта и создает основу для дальнейшего расширения функциональности.

# 4.2. Реализация логики регистрации и заполнения анкеты

Одним из ключевых сценариев работы ВК-бота ГоПати является регистрация пользователя и последовательное заполнение его анкеты. Данный процесс реализован как пошаговый диалоговый сценарий, в рамках которого бот последовательно запрашивает обязательные данные, контролирует корректность ввода и сохраняет результат в базе данных.

Логика регистрации построена на основе конечного набора состояний диалога. В проекте используются состояния «REG_NAME», «REG_AGE», «REG_GENDER», «REG_CITY», «REG_LOOKING», «STATE_GAMES», «STATE_REG_MICROPHONE», «STATE_ABOUT», «STATE_PHOTO» и промежуточные состояния, связанные с добавлением дополнительных фотографий. Каждое состояние соответствует конкретному этапу заполнения анкеты, а переходы между ними определяются действиями пользователя и результатами валидации введенных данных.

На этапе регистрации пользователь указывает имя, возраст, пол, город, цель поиска, выбирает интересующие игры, указывает, играет ли он с микрофоном, добавляет текстовое описание и фотографии. Для текстовых полей реализована валидация длины и типа содержимого, для возраста проверяется допустимый числовой диапазон, а для фотографий контролируется количество загружаемых изображений. Это позволяет обеспечить единообразие анкет и снизить число ошибок при вводе данных.

Дополнительной особенностью реализации является частичное автозаполнение анкеты из открытых данных профиля ВКонтакте. Если пользователь еще не ввел имя, город, пол или возраст, а эти сведения доступны через VK API, бот может использовать их как значения по умолчанию. Такой подход сокращает количество ручных действий и делает первичное заполнение анкеты более удобным.

Выбор игр реализован через inline-кнопки, а не через свободный текстовый ввод. Пользователь отмечает интересующие его позиции из заранее определенного справочника игр, после чего выбор сохраняется в связующей таблице «user_games». Факт заполнения игрового блока определяется наличием хотя бы одной выбранной игры, поэтому отдельный технический флаг для этого этапа не требуется.

Новая анкета по умолчанию создается неактивной: поле «profiles.is_active» имеет значение «0». Пока пользователь не заполнил все обязательные сведения, такая анкета хранится в базе как черновик и не попадает в выдачу другим пользователям. После прохождения всех обязательных шагов бот публикует анкету, устанавливая «is_active = 1». Такой подход исключает ситуацию, при которой незавершенная анкета может быть показана в подборе.

Фотографии сохраняются в таблице «user_photos». Для каждой фотографии хранится локальный путь «photo_path», порядок отображения «sort_order» и технический токен «vk_photo_token». При отправке анкеты бот сначала пытается использовать сохраненный VK-токен. Если он недоступен, фотография загружается с сервера по локальному пути, после чего новый токен сохраняется в базе для последующих отправок.

После завершения регистрации бот показывает пользователю экран собственной анкеты. На этом этапе пользователь может подтвердить корректность данных, перейти к просмотру анкет, открыть редактирование профиля или временно отключить анкету от участия в подборе. Таким образом, регистрационный сценарий не только собирает данные, но и переводит анкету из состояния черновика в состояние опубликованного профиля.

# 4.3. Реализация механизма просмотра анкет и истории взаимодействий

Механизм просмотра анкет реализует основной пользовательский сценарий сервиса: подбор подходящих профилей, отображение информации о кандидатах и фиксацию действий пользователя. После завершения регистрации пользователь может перейти к просмотру анкет, при этом бот выбирает только те профили, которые являются активными и полностью заполненными.

В выдачу попадают анкеты с признаком «profiles.is_active = 1». Дополнительно при подборе проверяется, что анкета не заблокирована, пользователь не отключен от доставки сообщений, профиль не принадлежит самому просматривающему пользователю, а также что в анкете есть обязательные данные, выбранные игры, фотографии и настройки поиска. Это необходимо для того, чтобы пользователь видел только доступные и завершенные анкеты, а черновики, заблокированные профили и пользователи, которым бот не может отправить сообщение, не участвовали в подборе.

Подбор кандидатов выполняется на основе данных из нормализованных таблиц. Основные сведения анкеты берутся из «profiles», выбранные игры - из «user_games», фотографии - из «user_photos», а параметры поиска - из «user_filters» и «user_filter_games». Такой подход позволяет не дублировать данные в одной большой таблице и сохраняет гибкость при развитии фильтров.

История действий пользователя хранится в таблице «interactions». Если пользователь уже поставил кандидату лайк или дизлайк, эта анкета не показывается повторно в обычном режиме просмотра. Для режима истории используются записи из той же таблицы: бот может вернуть пользователя к ранее оцененным анкетам и показать, какое действие было выполнено.

Для навигации по истории используются служебные поля runtime-состояния, включая «browse_mode», «history_candidate_action», «history_cursor_id» и «history_cursor_created_at». Эти данные не являются постоянными анкетными сведениями, а описывают текущий сценарий диалога. Они позволяют восстанавливать позицию пользователя в истории и поддерживать переходы между просмотром новых анкет и ранее выполненными действиями.

При отправке лайка бот сохраняет действие в таблице «interactions» и сразу отправляет получателю уведомление с анкетой отправителя. Если получатель отвечает лайком, в таблице появляется встречная запись, после чего система определяет взаимное совпадение. Если получатель отвечает дизлайком, это также фиксируется как действие пользователя и исключает повторный показ этой анкеты в обычном режиме просмотра.

Таким образом, просмотр анкет, история взаимодействий и обработка входящих лайков опираются на единый источник данных - таблицу «interactions». Это уменьшает дублирование данных и упрощает поддержку логики просмотра.

# 4.4. Реализация функций лайков, лайков с сообщением, мэтчей и жалоб

Сценарии пользовательского взаимодействия в ГоПати строятся вокруг оценки анкет и реакции на интересующие профили. В рамках проекта были реализованы функции обычного лайка, лайка с сообщением, дизлайка, обработки взаимных совпадений и подачи жалоб на анкеты.

Все пользовательские действия по отношению к другим анкетам сохраняются в таблице «interactions», которая выступает единым журналом взаимодействий между пользователями. Обычный лайк сохраняется как запись направления «from_user_id -> to_user_id» с действием «like». Дизлайк сохраняется аналогично, но со значением «dislike». Уникальное ограничение по паре «from_user_id» и «to_user_id» не позволяет хранить несколько актуальных действий одного пользователя по отношению к одной и той же анкете.

При постановке обычного лайка бот сохраняет действие в таблице «interactions» и проверяет, существует ли встречный лайк со стороны второго пользователя. Если взаимная симпатия подтверждается, бот определяет мэтч по двум встречным лайкам и отправляет обеим сторонам уведомления о взаимном совпадении. Если ответного лайка еще нет, получателю отправляется уведомление с анкетой пользователя, который проявил симпатию.

Отдельно реализован сценарий лайка с сообщением. В этом случае пользователь сначала инициирует действие, после чего бот переводит его в специальное состояние ввода текста. Сообщение ограничивается по длине и сохраняется в поле «interactions.like_message» вместе с действием «like». При последующем показе уведомления получателю бот может отобразить не только анкету отправителя, но и прикрепленное сообщение. Такой механизм делает общение менее формальным и позволяет пользователю сразу задать тон дальнейшему взаимодействию.

Дизлайк фиксируется как отдельный тип взаимодействия и используется для исключения нерелевантных анкет из дальнейшей выдачи. После постановки дизлайка бот, как правило, сразу переходит к следующей анкете, не заставляя пользователя выполнять дополнительные действия.

Сценарий входящих лайков также встроен в логику приложения. После получения уведомления о симпатии пользователь может ответить лайком, лайком с сообщением, дизлайком или жалобой. Ответ сохраняется в таблице «interactions» как встречное действие. Если пользователь отвечает лайком, возникает взаимное совпадение; если отвечает дизлайком или жалобой, анкета исключается из дальнейшего просмотра для этого пользователя.

Для обеспечения базовой безопасности и управляемости в проекте реализован механизм жалоб. Пользователь может указать причину жалобы на активную анкету, после чего бот отправляет сформированное сообщение в чат модерации. В сообщение включаются данные о профиле нарушителя, сведения о жалующемся пользователе и текст причины обращения. Дополнительно по отношению к такой анкете сохраняется отрицательное взаимодействие «dislike», чтобы она не продолжала показываться в выдаче.

Таким образом, функции лайков, лайков с сообщением, мэтчей и жалоб формируют основу прикладной ценности ГоПати. Они обеспечивают не только поиск потенциально интересных пользователей, но и управляемое развитие взаимодействия, включая переход от первичного выбора к взаимному знакомству и дальнейшему общению.

# 4.5. Реализация хранения данных и runtime-состояния пользователей

Корректная работа диалогового ВК-бота невозможна без надежного хранения как постоянных данных, так и временного состояния пользовательского сценария. В проекте ГоПати данная задача решена за счет сочетания реляционной базы данных MySQL и in-memory-представления текущего состояния пользователя в серверном процессе.

Постоянные данные системы хранятся в таблицах «users», «profiles», «games», «user_games», «user_photos», «interactions», «user_sessions», «user_filters» и «user_filter_games». Через эти сущности фиксируются анкеты пользователей, их игровые интересы, фотографии, история действий, фильтры поиска и служебные данные, необходимые для логики приложения.

Таблица «profiles» хранит основные сведения анкеты: имя, возраст, город, описание, пол, признак использования микрофона, активность профиля и служебные признаки блокировки или недоступности доставки сообщений. Настройки поиска хранятся в таблице «user_filters», поскольку они относятся не к описанию самой анкеты, а к параметрам подбора других пользователей.

Игровые интересы пользователя хранятся в таблице «user_games», которая связывает пользователя со справочником «games». В этой таблице фиксируется соответствие между пользователем и выбранной игрой через поля «user_id» и «game_id». Такой формат отражает связь «многие ко многим» и позволяет пользователю выбирать несколько игр.

Фотографии анкеты хранятся в таблице «user_photos». Для каждой фотографии сохраняется локальный путь «photo_path», технический токен «vk_photo_token» и порядок отображения «sort_order». При отправке анкеты бот сначала пытается использовать сохраненный VK-токен, а при его недоступности загружает фотографию с сервера по локальному пути и сохраняет новый токен для последующих отправок.

Таблица «interactions» используется как единое хранилище пользовательских действий. В ней фиксируются лайки, дизлайки и сообщения к лайкам. Взаимное совпадение определяется по двум встречным лайкам, а история просмотра строится на основе уже сохраненных действий пользователя. Такой подход уменьшает дублирование данных и сохраняет в базе первичные действия, из которых можно получить производные состояния.

Отдельное значение имеет таблица «user_sessions», в которой хранится сериализованное состояние диалога пользователя в формате JSON. В поле «session_json» сохраняются текущий шаг сценария, просматриваемая анкета, режим просмотра, курсоры истории, временные значения для отправки лайка с сообщением, данные для жалобы и другие служебные признаки. За счет этого бот способен продолжить диалог с нужного места после повторного обращения пользователя или после перезапуска приложения.

Во время работы приложения актуальное состояние пользователя также присутствует в памяти процесса в глобальном runtime-хранилище. При новом обращении пользователя происходит синхронизация между оперативным представлением и данными, сохраненными в базе. Такой механизм позволяет одновременно обеспечить высокую скорость работы диалога и устойчивость к перезапуску серверной части.

Для загрузки и восстановления состояния используется специализированная логика, объединяющая данные профиля, выбранные игры, фотографии, пользовательскую сессию и часть открытых сведений профиля ВК. Это позволяет не только восстановить технический контекст взаимодействия, но и актуализировать представление анкеты пользователя в рамках текущего диалога.

Дополнительно в проекте предусмотрена runtime-подготовка схемы БД через процедуру «ensure_runtime_schema()». Она проверяет наличие необходимых таблиц и служебных колонок, связанных с модерацией, доставкой сообщений, пользовательскими сессиями и хранением взаимодействий. Такой подход делает систему более устойчивой при сопровождении и снижает риск ошибок, связанных с несовпадением схемы данных и прикладной логики.

В процессе функционирования системы осуществляется обработка персональных данных пользователей, включая имя, возраст, город проживания, фотографии и сведения об игровых интересах. Обработка указанных данных выполняется исключительно в целях обеспечения работы сервиса и предоставления пользователям возможности поиска игровых напарников и новых контактов. В соответствии с требованиями Федерального закона №152-ФЗ «О персональных данных» пользователь самостоятельно предоставляет сведения при заполнении анкеты и может временно отключить свой профиль от участия в подборе анкет с помощью предусмотренных механизмов системы. При необходимости полного удаления данных пользователь может обратиться к администратору сервиса, после чего соответствующие записи могут быть удалены из базы данных. Передача пользовательских данных третьим лицам не осуществляется, а взаимодействие с платформой «ВКонтакте» выполняется через защищенное HTTPS-соединение.

Таким образом, механизм хранения данных в ГоПати объединяет постоянное реляционное хранение и временное runtime-состояние, что обеспечивает устойчивость работы бота, целостность пользовательских сценариев и возможность восстановления контекста взаимодействия после перезапуска системы.

# 4.6. Настройка окружения, базы данных и запуск приложения

Для обеспечения круглосуточной работы ВК-бота ГоПати была выполнена настройка удаленной серверной инфраструктуры [8]. В качестве площадки размещения использовался облачный сервис «Cloud.ru Evolution», в котором был арендован виртуальный сервер стоимостью «150 руб. в месяц». При развертывании был выбран образ «Ubuntu 22.04» со следующими параметрами: загрузочный диск «30 ГБ», гарантированная доля вычислительных ресурсов «10%», «2 vCPU» и «4 ГБ RAM» [22]. Данная конфигурация является достаточной для размещения серверной части бота, reverse proxy, базы данных и вспомогательных компонентов (рис. 39).

Рисунок 39 - Параметры виртуального сервера в Cloud.ru, используемого для размещения ВК-бота «ГоПати»

Для обеспечения работы «ВК Callback API» потребовалось наличие публично доступного домена и HTTPS-соединения. С этой целью был зарегистрирован домен «gopartybot.ru» у регистратора «Reg.ru» стоимостью 169 руб. в год. В DNS-настройках домена были созданы «A»-записи для корневого домена и поддомена «www», указывающие на публичный IP-адрес сервера. Это позволило связать доменное имя с развернутым приложением и использовать единый адрес для обращения со стороны платформы «ВКонтакте» (рис. 40).

Рисунок 40 - DNS-настройки домена gopartybot.ru в панели управления Reg.ru

На сервере была развернута серверная часть приложения на базе «FastAPI», а в качестве reverse proxy использовался «nginx» [20]. HTTP-запросы, поступающие на домен, проксируются во внутреннее приложение по маршруту «https://gopartybot.ru/vk/callback». Такая схема позволила изолировать внутренний Python-процесс от прямого внешнего доступа и обеспечить удобное управление сетевой конфигурацией.

Для открытия доступа к приложению на уровне сетевой безопасности были созданы два TCP-правила межсетевого экрана: для порта «80:80» с описанием «HTTP» и для порта «443:443» с описанием «HTTPS», оба с источником «0.0.0.0/0». Эти правила обеспечили прием обычного HTTP-трафика и защищенного HTTPS-трафика извне (рис. 41).

Рисунок 41 - Правила входящего и исходящего трафиков виртуального сервера

Для выпуска TLS-сертификата был установлен «certbot» - программный инструмент для автоматического получения, установки и продления SSL/TLS-сертификатов от центра сертификации «Let’s Encrypt».

Был выпущен сертификат для домена и поддомена:

```bash
sudo certbot --nginx -d gopartybot.ru -d www.gopartybot.ru
```

Выпуск сертификата позволил перевести приложение на HTTPS, что является обязательным условием для работы callback-сервера «ВКонтакте» в публичной среде. После получения сертификата «nginx» был настроен на обслуживание защищенного соединения и проксирование запросов к приложению «FastAPI» (рис. 42).

После запуска приложения был настроен endpoint здоровья «/health», а также обеспечен прием событий «message_new» и «message_event» от платформы ВК. В настройках сообщества «ВКонтакте» callback URL был указан в виде «https://gopartybot.ru/vk/callback» (рис. 43).

Для подготовки базы данных использовалась SQL-миграция «migration_20260610_normalize_interactions.sql». Она приводит структуру БД к рабочей схеме проекта: настраивает таблицу «interactions» для хранения лайков, дизлайков и сообщений к лайкам, размещает настройки поиска в «user_filters», а также задает корректные значения по умолчанию для публикации анкет.

Для поля «profiles.is_active» используется значение по умолчанию «0». Благодаря этому новая анкета создается как черновик и становится видимой другим пользователям только после полного заполнения обязательных данных. Для снижения риска потери данных предусматривается выполнение резервного копирования базы данных перед внесением изменений в структуру БД и выполнением операций массовой обработки данных. В случае возникновения критических ошибок данные могут быть восстановлены из резервной копии, что позволяет сократить время простоя системы и минимизировать потерю пользовательской информации.

Таким образом, этап настройки окружения включал не только запуск Python-приложения, но и создание полноценной удаленной инфраструктуры: аренду облачного сервера, регистрацию домена, выпуск SSL-сертификата, настройку сетевых правил, интеграцию «nginx» и «FastAPI», подготовку базы данных, а также настройку окружения для приема callback-событий от платформы «ВКонтакте».

# 4.7. Тестирование и проверка работоспособности системы

После реализации основных компонентов ВК-бота ГоПати была проведена проверка работоспособности системы. Цель тестирования заключалась в подтверждении корректной работы ключевых пользовательских сценариев и устойчивости серверной части.

Следует отметить, что в рамках данной работы проводилось функциональное и интеграционное тестирование основных пользовательских сценариев. Нагрузочное тестирование, оценка производительности при большом количестве одновременных пользователей, измерение показателей RPS и исследование отказоустойчивости не выполнялись, поскольку разработанное решение рассматривается как proof of concept. Проведение подобных испытаний целесообразно на этапе подготовки системы к промышленной эксплуатации.

На первом этапе была выполнена проверка запуска серверного приложения и корректности обработки callback-запросов. Для этого контролировалась доступность endpointов «/health» и «/vk/callback», а также проверялась корректность прохождения события подтверждения сервера от платформы «ВКонтакте». Дополнительно анализировалась работа reverse proxy «nginx» и доступность приложения по защищенному адресу «https://gopartybot.ru/vk/callback».

На втором этапе тестировались пользовательские сценарии регистрации и заполнения анкеты. Проверялась последовательность переходов между шагами диалога, корректность валидации имени, возраста, города, описания и фотографий, а также сохранение выбранных игр. Дополнительно проверялась возможность частичного автозаполнения анкеты на основе открытых данных профиля ВК.

Отдельно проверялось, что новая анкета создается со значением «is_active = 0» и не попадает в выдачу до завершения регистрации. После заполнения всех обязательных данных проверялось изменение значения «is_active» на «1», что подтверждало публикацию анкеты и ее доступность для показа другим пользователям.

Далее была выполнена проверка сценариев просмотра анкет и пользовательских взаимодействий. Тестировались показ новых профилей, исключение уже оцененных анкет, корректная работа лайков, дизлайков, лайков с сообщением, формирование уведомления о входящем лайке и создание мэтча при взаимной симпатии. Особое внимание уделялось корректности переходов между режимами просмотра и отсутствию повторного показа уже обработанных анкет.

Для лайков с сообщением проверялось сохранение текста в поле «interactions.like_message». Для мэтчей проверялось, что взаимное совпадение определяется по двум встречным лайкам в «interactions». Для входящих лайков проверялась отправка уведомления получателю и сохранение его ответа как встречного действия.

Отдельно проверялась работа фотографий. При отправке анкеты бот сначала использует «vk_photo_token»; если токен становится недоступным, фото загружается из локального хранилища по «photo_path», после чего новый VK-токен сохраняется в базе. Такой сценарий позволяет сохранять работоспособность отправки фотографий даже при устаревании токенов VK.

Отдельно проверялась работа механизмов модерации и устойчивости. В рамках тестирования анализировалась отправка жалобы в чат модерации, обработка деактивации анкеты, полный сброс пользовательских данных, очистка истории, а также восстановление диалога после перезапуска приложения. Для этого оценивалась корректность сохранения runtime-состояния в таблице «user_sessions» и возврат пользователя к нужному сценарию после повторного обращения к боту.

В процессе проверки также использовались технические журналы «logs/bot.log» и «logs/actions.jsonl», позволившие контролировать входящие события, ошибки обработки и продолжительность выполнения отдельных операций. Анализ журналов дал возможность выявлять проблемные участки, в том числе ошибки отправки фотографий через VK API, и подтверждать успешную обработку событий «message_new» и «message_event».

Результаты тестирования показали, что реализованная система выполняет основные функции в соответствии с поставленными требованиями: принимает callback-события от «ВКонтакте», корректно ведет пользователя по диалогу, сохраняет данные в базе, поддерживает восстановление состояния и обеспечивает сценарии поиска, оценки анкет и взаимного знакомства между пользователями.
