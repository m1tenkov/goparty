# ГЛАВА 2. УПРАВЛЕНИЕ ПРОЕКТОМ

## 2.1. Оценка рисков

Разработка ВК-бота «ГоПати» связана не только с реализацией программной логики, но и с учетом факторов, которые могут повлиять на стабильность работы сервиса, сроки выполнения проекта и удобство дальнейшей эксплуатации. Поскольку система работает внутри социальной сети «ВКонтакте», принимает callback-события, хранит анкеты пользователей и обеспечивает подбор людей по игровым интересам, особое значение имеют риски, связанные с внешней платформой, корректностью диалоговых сценариев, сохранностью данных и модерацией пользовательского контента.

В рамках проекта можно выделить несколько групп рисков. Технические риски связаны с ошибками серверной части, работой базы данных MySQL, обработкой состояний пользователя и доступностью удаленной инфраструктуры. Интеграционные риски возникают из-за зависимости от VK API и правил платформы «ВКонтакте». Пользовательские риски относятся к качеству анкет, жалобам, неподходящему контенту и снижению интереса аудитории к формату бота. Организационные риски связаны с ограниченным количеством исполнителей, сроками тестирования и необходимостью совмещать роли аналитика, разработчика, тестировщика и администратора проекта.

Для оценки использовались два показателя: величина угрозы и вероятность наступления риска. Величина угрозы задается по шкале от 1 до 10, где 1 соответствует незначительному влиянию, а 10 означает критическое воздействие на проект. Вероятность указывается в процентах. Ожидаемая величина риска рассчитывается как произведение величины угрозы на вероятность, выраженную в долях единицы. Перечень наиболее значимых рисков представлен в таблице 2.

По результатам оценки наибольшего внимания требуют риски, связанные со сбоями обмена данными между ботом и платформой «ВКонтакте», потерей пользовательских данных и недостаточной модерацией пользовательского контента. Эти угрозы затрагивают базовые условия работы проекта: возможность получать события от пользователей, отправлять ответы, сохранять анкеты, поддерживать доверие аудитории и обеспечивать безопасное взаимодействие внутри сервиса.

Для подробного анализа наиболее существенных угроз были подготовлены отдельные паспорта рисков, позволяющие зафиксировать причины их возникновения, возможные последствия и меры реагирования (таблицы 3-5).

Таблица 3. Паспорт риска «Сбои обмена данными между ботом и платформой ВКонтакте»

| Показатель | Описание |
| --- | --- |
| Название | Сбои обмена данными между ботом и платформой ВКонтакте |
| Формулировка | Нарушение обмена данными между серверной частью ВК-бота и платформой «ВКонтакте» при получении callback-событий, отправке сообщений или обработке действий пользователя |
| Последствия | Пользователь может не получить ответ от бота, уведомление о лайке или мэтче может не отправиться, а отдельные сценарии регистрации и просмотра анкет могут завершиться с ошибкой |
| Причинно-следственные связи | Работа сервиса зависит от корректной настройки callback-endpoint, токена сообщества, доступности VK API, структуры входящих событий и обработки ошибок внешней платформы. При сбоях в любом из этих элементов нарушается пользовательский сценарий |
| Вероятность | 70% |
| Величина ущерба | 5,6 |
| Предотвращение риска | Предусмотреть отдельную точку приема callback-событий, проверку секретного ключа, раздельную обработку входящих сообщений и нажатий на inline-кнопки, а также обработку ситуаций, когда сообщение пользователю не может быть доставлено |
| Минимизация последствий | При возникновении сбоя необходимо проверить настройки callback-сервера и токена, зафиксировать ошибку в технических логах, исправить обработчики событий, а при ошибках доставки пометить профиль как недоступный для выдачи до следующего обращения пользователя к боту |
| Форма текущей отчетности | Ошибки и признаки нестабильной работы фиксировать в технических логах |
| Владелец | Backend-разработчик |

Таблица 4. Паспорт риска «Потеря или повреждение пользовательских данных»

| Показатель | Описание |
| --- | --- |
| Название | Потеря или повреждение пользовательских данных |
| Формулировка | Нарушение целостности или доступности данных, хранящихся в базе данных или на сервере, включая анкеты, фотографии, игровые интересы, лайки, жалобы, историю просмотров и runtime-состояния пользователей |
| Последствия | Пользователи могут потерять заполненные анкеты, система подбора начнет работать некорректно, восстановление диалога после перезапуска станет проблематичным, а доверие к сервису снизится |
| Причинно-следственные связи | Данные проекта хранятся в реляционной базе и изменяются при большинстве пользовательских действий. Ошибки в SQL-запросах, сбои сервера, неудачные изменения структуры таблиц или отсутствие резервных копий могут привести к повреждению записей |
| Вероятность | 40% |
| Величина ущерба | 3,2 |
| Предотвращение риска | Предусмотреть нормализованную структуру базы данных, проверку SQL-запросов, ограничение ручного вмешательства в рабочие таблицы и создание резервных копий перед изменением схемы или массовой обработкой данных |
| Минимизация последствий | При возникновении проблемы выполнить восстановление данных из резервной копии, повторную синхронизацию пользовательских состояний, временное отключение сценариев, которые могут усугубить ошибку |
| Форма текущей отчетности | Технические логи приложения, структурированный журнал действий «logs/actions.jsonl», контрольные выборки по таблицам в MySQL/DBeaver и фиксация ручных изменений схемы. |
| Владелец | Backend-разработчик |

Таблица 5. Паспорт риска «Недостаточная модерация пользовательского контента»

| Показатель | Описание |
| --- | --- |
| Название | Недостаточная модерация пользовательского контента |
| Формулировка | Возникновение ситуации, при которой жалобы, спам, неподходящие фотографии или некорректные описания анкет обрабатываются недостаточно быстро либо не выявляются на раннем этапе |
| Последствия | Ухудшение пользовательского опыта, рост количества жалоб, снижение доверия к сервису, появление негативного впечатления о проекте и возможное прекращение использования бота частью аудитории |
| Причинно-следственные связи | Сервис знакомств предполагает публикацию пользовательских анкет и фотографий, а также взаимодействие между незнакомыми людьми. При отсутствии достаточного контроля отдельные пользователи могут размещать неподходящий контент, отправлять спам или нарушать правила общения |
| Вероятность | 40% |
| Величина ущерба | 2,8 |
| Предотвращение риска | Предусмотреть механизм отправки жалобы в чат модерации, фиксацию отрицательного взаимодействия с анкетой и возможность ручной блокировки нарушителей через административные SQL-скрипты |
| Минимизация последствий | При выявлении нарушения вручную проверить обращение, при необходимости заблокировать профиль через служебный SQL-скрипт, удалить неподходящие данные из БД/хранилища и исключить анкету из дальнейшей выдачи |
| Форма текущей отчетности | Сообщения о жалобах в чате модерации, записи «logs/actions.jsonl», технический лог «logs/bot.log» |
| Владелец | Модератор, руководитель проекта, backend-разработчик |

Проведенная оценка показывает, что для ВК-бота «ГоПати» наиболее существенные угрозы сосредоточены вокруг корректности обмена данными с «ВКонтакте», сохранности пользовательской информации и контроля качества пользовательского контента. Эти риски не ограничиваются отдельными техническими дефектами, а могут повлиять на стабильность эксплуатации сервиса и доверие аудитории. Поэтому при дальнейшем развитии проекта необходимо поддерживать логирование, резервное копирование, проверку основных пользовательских сценариев, обработку жалоб. Такой подход позволяет заранее определить меры реагирования и снизить вероятность критических сбоев при использовании системы.
