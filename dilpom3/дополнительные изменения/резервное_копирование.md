# Резервное копирование данных

В рамках доработки проекта был добавлен механизм резервного копирования данных бота. Данная доработка направлена на повышение надежности системы и снижение риска потери пользовательских данных при сбоях, ошибках обновления или повреждении базы данных.

Резервное копирование реализовано отдельным скриптом `scripts/backup.py`, который запускается на сервере Cloud.ru, где размещено приложение. Скрипт формирует единый архив резервной копии, включающий два основных источника данных:

- SQL-дамп базы данных MySQL `goparty_bot`;
- локальные файлы пользовательских фотографий из каталога `storage/photos`.

Такой подход важен, потому что в базе данных хранятся анкеты, настройки, история взаимодействий, лайки, мэтчи и служебные состояния пользователей, а фотографии физически находятся в файловом хранилище сервера. Если сохранять только базу данных без файлов, после восстановления часть анкет могла бы ссылаться на отсутствующие изображения.

Скрипт создает архив формата ZIP с именем вида:

```text
goparty_backup_YYYYMMDD_HHMMSS.zip
```

Внутри архива сохраняются:

- `database.sql` - резервная копия структуры и данных MySQL;
- `storage/photos` - каталог с фотографиями пользователей;
- `manifest.json` - служебная информация о времени создания копии и источниках данных.

Для автоматического выполнения резервного копирования предусмотрен запуск скрипта по расписанию на сервере. Планируется ежедневное создание резервной копии в ночное время, например в 03:15, когда нагрузка на бота минимальна:

```cron
15 3 * * * cd /path/to/project && /path/to/venv/bin/python scripts/backup.py --output-dir /var/backups/goparty --keep 30 >> logs/backup.log 2>&1
```

Параметр `--keep 30` ограничивает количество хранимых копий: на сервере остаются 30 последних архивов, а более старые удаляются автоматически. Это позволяет не переполнять дисковое пространство и при этом иметь историю восстановления примерно за последний месяц.

Дополнительно предусмотрена возможность переноса архивов за пределы основного сервера, например на локальный компьютер администратора или во внешнее объектное хранилище. Это повышает надежность, так как копии остаются доступными даже при потере или повреждении виртуальной машины.

Таким образом, резервное копирование реализовано как отдельный эксплуатационный механизм, не влияющий на работу пользовательских сценариев бота. Оно выполняется автоматически по расписанию, сохраняет как базу данных, так и пользовательские файлы, а также поддерживает ограничение срока хранения старых копий.
